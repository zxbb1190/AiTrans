# 知识库接口框架标准模块:KnowledgeBaseApiFramework

@framework

本模块在 `Framework -> Product Spec -> Implementation Config -> Code` 分层中只负责接口共同结构语言与稳定契约边界；具体产品对外暴露什么接口语义应先收敛为 `Product Spec`，再由 `Implementation Config` 细化为某种实现路径，本模块不得直接固化库名、运行时或项目专属实现细节。对应的 `product_spec.toml` 与 `implementation_config.toml` 应默认使用中文详细注释，以便接口语义、实现承载与验证出口在同一套仓库语言里被清楚讨论。

## 1. 能力声明（Capability Statement）

- `C1` 文件管理接口提供能力：稳定提供文件列表、上传、移除与文件聚焦相关接口。
- `C2` 文档预览接口提供能力：稳定提供正文、目录与章节锚点相关接口。
- `C3` 对话引用接口提供能力：稳定提供基于知识库的对话请求、回答输出与引用返回接口。
- `C4` 非能力项：不负责前端布局、检索算法优化与模型回答质量定义。

## 2. 边界定义（Boundary / 参数）

- `LIBRARY` 文件库边界：文件列表、上传、移除与文件聚焦接口需满足统一约束。来源：`C1 + C3`。
- `PREVIEW` 预览边界：正文、目录与章节锚点接口需返回完整预览结构。来源：`C1 + C2`。
- `CHAT` 对话边界：输入消息、回答消息、引用域与引用条目结构需统一。来源：`C2 + C3`。
- `RESULT` 结果边界：文件、预览与对话接口需共享统一外层返回结构。来源：`C1 + C2 + C3`。
- `AUTH` 权限边界：写入类接口必须显式声明权限约束或替代策略。来源：`C1 + C3`。
- `TRACE` 追踪边界：请求参数、错误原因与引用来源必须可追踪。来源：`C2 + C3`。

## 3. 最小可行基（Minimum Viable Bases）

- `B1` 文件预览接口基：L1.M0[R1,R3]。来源：`C1 + C2 + LIBRARY + PREVIEW + RESULT`。
- `B2` 对话引用接口基：L1.M0[R2,R3]。来源：`C3 + CHAT + RESULT + TRACE`。
- `B3` 接口治理基：L1.M0[R1,R2,R3]。来源：`RESULT + AUTH + TRACE`。

## 4. 基组合原则（Base Combination Principles）

- `R1` 文件预览能力装配
  - `R1.1` 参与基：`B1`。
  - `R1.2` 组合方式：先固定文件管理接口与文档预览接口的参数和返回结构，形成稳定文件预览面。
  - `R1.3` 输出能力：`C1 + C2`。
  - `R1.4` 边界绑定：`LIBRARY + PREVIEW + RESULT`。
- `R2` 对话引用能力装配
  - `R2.1` 参与基：`B2 + B3`。
  - `R2.2` 组合方式：在统一治理约束下接入对话输入、回答输出与引用返回结构。
  - `R2.3` 输出能力：`C2 + C3`。
  - `R2.4` 边界绑定：`CHAT + TRACE + RESULT + AUTH`。
- `R3` 知识库接口闭环
  - `R3.1` 参与基：`B1 + B2 + B3`。
  - `R3.2` 组合方式：把文件管理、文档预览与对话引用接口收敛为可直接承接知识库工作台的稳定接口面。
  - `R3.3` 输出能力：`C1 + C2 + C3`。
  - `R3.4` 边界绑定：`LIBRARY + PREVIEW + CHAT + RESULT + AUTH + TRACE`。
- `R4` 禁止组合
  - `R4.1` 参与基：`B1 + B2 + B3`。
  - `R4.2` 组合方式：结果结构不一致、引用来源不可回看或写入接口无治理约束的方案均无效。
  - `R4.3` 输出能力：`C4`。
  - `R4.4` 边界绑定：`LIBRARY + PREVIEW + CHAT + RESULT + AUTH + TRACE`。

## 5. 验证（Verification）

- `V1` 文件管理覆盖性：列表、上传与移除接口必须全部存在且结构稳定。
- `V2` 预览覆盖性：正文、目录与章节锚点接口必须满足 `PREVIEW` 约束。
- `V3` 对话引用覆盖性：对话接口必须返回稳定引用来源结构。
- `V4` 契约一致性：所有接口必须满足统一 `RESULT + AUTH + TRACE` 约束。
- `V5` 最小必要性：移除任一 `B*` 后，`V2~V4` 至少一项失败。
- `V6` 结论表达：逐条输出 `R* -> C* + 边界绑定` 的通过或失败结论。
