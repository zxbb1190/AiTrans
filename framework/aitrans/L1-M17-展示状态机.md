# 展示状态机:ZhanShiZhuangTaiJi

@framework

## 1. 能力声明（Capability Statement）

- `C1` 状态建模能力：提供展示态、加载态、成功态、失败态、空态与关闭态建模能力。
- `C2` 状态流转能力：提供展示状态切换、合法流转校验与回退控制能力。
- `C3` 状态驱动能力：提供视图渲染、交互启停与显隐行为的状态驱动能力。
- `C4` 异常状态收敛能力：提供错误态、部分成功态与恢复态收敛能力。
- `C5` 状态事件通知能力：提供状态变更事件广播与依赖同步能力。
- `C6` 非能力项：不负责具体 UI 绘制与动作执行细节。

## 2. 边界定义（Boundary / 参数）

- `MODEL` 状态模型约束：展示状态必须具备完整状态集合与明确语义。来源：`C1`。
- `TRANSITION` 流转约束：状态切换必须满足合法流转规则。来源：`C2`。
- `DRIVE` 驱动约束：状态变化必须驱动视图、交互与显隐联动。来源：`C3`。
- `RECOVER` 恢复约束：异常状态必须可被收敛到可恢复或终止状态。来源：`C4`。
- `EVENT` 事件约束：状态变更必须可广播与被依赖模块消费。来源：`C5`。
- `CONSISTENCY` 一致性约束：状态、视图与交互行为必须保持一致。来源：`C1 + C2 + C3 + C4 + C5`。

## 3. 最小可行基（Minimum Viable Bases）

- `B1` 状态模型骨架：L0.M25[展示状态流转器]。来源：`C1 + C2 + MODEL + TRANSITION`。
- `B2` 驱动恢复骨架：L0.M26[展示驱动恢复器]。来源：`C3 + C4 + DRIVE + RECOVER`。
- `B3` 广播审计骨架：L0.M3[事件总线] + L0.M9[审计追踪器]。来源：`C5 + EVENT + CONSISTENCY`。

## 4. 基组合原则（Base Combination Principles）

- `R1` 状态模型组合
  - `R1.1` 参与基：`B1`。
  - `R1.2` 组合方式：先定义展示状态集合、初始状态与合法流转规则，形成统一状态源。
  - `R1.3` 输出能力：`C1 + C2`。
  - `R1.4` 边界绑定：`MODEL / TRANSITION`。

- `R2` 驱动恢复组合
  - `R2.1` 参与基：`B1 + B2`。
  - `R2.2` 组合方式：将状态源与视图驱动、交互启停、异常恢复收敛绑定，形成完整展示状态控制链路。
  - `R2.3` 输出能力：`C1 + C2 + C3 + C4`。
  - `R2.4` 边界绑定：`DRIVE / RECOVER / CONSISTENCY`。

- `R3` 广播审计组合
  - `R3.1` 参与基：`B1 + B2 + B3`。
  - `R3.2` 组合方式：将状态变化、驱动结果与恢复过程绑定到事件广播与审计追踪链路。
  - `R3.3` 输出能力：`C1 + C2 + C3 + C4 + C5`。
  - `R3.4` 边界绑定：`EVENT / CONSISTENCY`。

- `R4` 禁止组合
  - `R4.1` 参与基：`B1 + B2 + B3`。
  - `R4.2` 组合方式：无合法流转规则直接切换展示状态，或异常状态无驱动恢复与广播审计收敛的方案均无效。
  - `R4.3` 输出能力：`C6`。
  - `R4.4` 边界绑定：`TRANSITION / RECOVER / CONSISTENCY`。

## 5. 验证（Verification）

- `V1` 状态模型有效性：展示状态集合与合法流转规则必须完整。
- `V2` 驱动恢复有效性：状态变化必须正确驱动视图、交互与显隐行为，错误态与部分成功态必须具备恢复或终止收敛路径。
- `V3` 广播审计有效性：状态变化必须可广播并被依赖模块消费。
- `V4` 一致性符合性：状态、视图与交互必须满足 `CONSISTENCY` 约束。
- `V5` 最小必要性：移除任一 `B1~B3` 后，`V2~V4` 至少一项失败。
- `V6` 结论表达：逐条输出 `R* -> C* / 边界` 的通过或失败结论。
