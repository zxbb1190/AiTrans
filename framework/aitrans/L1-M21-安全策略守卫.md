# 安全策略守卫:AnQuanCeLveShouWei

@framework

## 1. 能力声明（Capability Statement）

- `C1` 权限最小化能力：提供渲染上下文能力收敛与最小权限控制能力。
- `C2` 接口访问控制能力：提供高风险接口、文件访问与系统能力访问控制能力。
- `C3` 依赖治理能力：提供第三方依赖风险识别、约束与升级治理能力。
- `C4` 内容安全能力：提供资源加载约束、脚本执行限制与内容安全策略能力。
- `C5` 敏感数据保护能力：提供密钥、配置与临时数据隔离保护能力。
- `C6` 非能力项：不负责具体业务数据处理逻辑。

## 2. 边界定义（Boundary / 参数）

- `PERMISSION` 权限约束：进程与模块能力必须按最小权限原则分配。来源：`C1`。
- `ACCESS` 访问约束：系统接口与高风险能力访问必须受控。来源：`C2`。
- `DEPENDENCY` 依赖约束：第三方依赖必须经过清单治理与风险限制。来源：`C3`。
- `CSP` 内容安全约束：资源加载与脚本执行必须符合内容安全策略。来源：`C4`。
- `SECRET` 敏感数据约束：密钥、配置与临时缓存不得不受控暴露。来源：`C5`。
- `AUDIT` 审计约束：关键安全决策必须可追踪与可审计。来源：`C1 + C2 + C3 + C4 + C5`。

## 3. 最小可行基（Minimum Viable Bases）

- `B1` 权限访问骨架：L0.M31[权限访问控制器]。来源：`C1 + C2 + PERMISSION + ACCESS`。
- `B2` 供应链内容骨架：L0.M32[供应链内容安全器]。来源：`C3 + C4 + DEPENDENCY + CSP`。
- `B3` 敏感数据骨架：L0.M33[密钥保护控制器]。来源：`C5 + SECRET + AUDIT`。

## 4. 基组合原则（Base Combination Principles）

- `R1` 权限访问组合
  - `R1.1` 参与基：`B1`。
  - `R1.2` 组合方式：先建立权限矩阵与高风险接口访问控制，形成最小权限访问面。
  - `R1.3` 输出能力：`C1 + C2`。
  - `R1.4` 边界绑定：`PERMISSION / ACCESS`。

- `R2` 供应链内容组合
  - `R2.1` 参与基：`B1 + B2`。
  - `R2.2` 组合方式：在最小权限访问面上叠加第三方依赖治理与资源加载、脚本执行策略，形成供应链与运行内容安全面。
  - `R2.3` 输出能力：`C1 + C2 + C3 + C4`。
  - `R2.4` 边界绑定：`DEPENDENCY / CSP / ACCESS`。

- `R3` 敏感数据组合
  - `R3.1` 参与基：`B1 + B2 + B3`。
  - `R3.2` 组合方式：将最小权限访问、供应链内容安全与密钥、配置、缓存隔离保护绑定，形成完整安全策略闭环。
  - `R3.3` 输出能力：`C1 + C2 + C3 + C4 + C5`。
  - `R3.4` 边界绑定：`SECRET / AUDIT / PERMISSION / ACCESS`。

- `R4` 禁止组合
  - `R4.1` 参与基：`B1 + B2 + B3`。
  - `R4.2` 组合方式：无最小权限访问骨架直接暴露系统能力，或无敏感数据骨架直接在渲染上下文保存密钥的方案均无效。
  - `R4.3` 输出能力：`C6`。
  - `R4.4` 边界绑定：`PERMISSION / ACCESS / SECRET`。

## 5. 验证（Verification）

- `V1` 权限访问有效性：权限矩阵与高风险接口、文件访问、系统能力访问必须受控。
- `V2` 供应链内容有效性：第三方依赖治理与内容安全策略必须满足 `DEPENDENCY / CSP` 约束。
- `V3` 敏感数据有效性：密钥、配置与临时数据必须被隔离保护与审计。
- `V4` 最小必要性：移除任一 `B1~B3` 后，`V1~V3` 至少一项失败。
- `V5` 结论表达：逐条输出 `R* -> C* / 边界` 的通过或失败结论。
