# 构建发布管理器:GouJianFaBuGuanLiQi

@framework

## 1. 能力声明（Capability Statement）

- `C1` 构建编排能力：提供开发、测试、打包与发布流程编排能力。
- `C2` 环境注入能力：提供构建环境、运行环境与发布环境变量注入能力。
- `C3` 产物管理能力：提供安装包、资源文件与版本产物管理能力。
- `C4` 版本治理能力：提供版本号管理、变更标识与发布序列控制能力。
- `C5` 更新策略能力：提供自动更新、手动更新与回退策略能力。
- `C6` 非能力项：不负责具体业务功能实现。

## 2. 边界定义（Boundary / 参数）

- `BUILD` 构建约束：构建流程必须支持开发、测试、打包与发布四类阶段。来源：`C1`。
- `ENV` 环境约束：环境变量与环境配置必须按阶段区分并受控注入。来源：`C2`。
- `ARTIFACT` 产物约束：构建产物必须可识别、可追踪、可校验。来源：`C3`。
- `VERSION` 版本约束：版本变更必须具备唯一标识与发布顺序约束。来源：`C4`。
- `UPDATE` 更新约束：更新策略必须支持检测、下载、安装与失败回退。来源：`C5`。
- `COMPAT` 兼容约束：不同平台构建与发布规则必须被统一治理。来源：`C1 + C3 + C4 + C5`。

## 3. 最小可行基（Minimum Viable Bases）

- `B1` 构建注入骨架：L0.M34[构建注入协调器]。来源：`C1 + C2 + BUILD + ENV`。
- `B2` 发布更新骨架：L0.M35[发布更新治理器]。来源：`C3 + C4 + C5 + ARTIFACT + VERSION + UPDATE + COMPAT`。

## 4. 基组合原则（Base Combination Principles）

- `R1` 构建注入组合
  - `R1.1` 参与基：`B1`。
  - `R1.2` 组合方式：先建立开发、测试、打包、发布阶段的流程编排与环境注入规则，形成统一构建底座。
  - `R1.3` 输出能力：`C1 + C2`。
  - `R1.4` 边界绑定：`BUILD / ENV`。

- `R2` 发布更新组合
  - `R2.1` 参与基：`B1 + B2`。
  - `R2.2` 组合方式：在构建底座上绑定产物清单、校验、版本发布顺序与更新检测、安装、回退策略，形成受控发布链路。
  - `R2.3` 输出能力：`C1 + C2 + C3 + C4 + C5`。
  - `R2.4` 边界绑定：`ARTIFACT / VERSION / UPDATE / COMPAT`。

- `R3` 禁止组合
  - `R3.1` 参与基：`B1 + B2`。
  - `R3.2` 组合方式：无环境隔离直接复用发布配置进行开发构建，或无发布更新骨架直接分发产物与暴露更新路径的方案均无效。
  - `R3.3` 输出能力：`C6`。
  - `R3.4` 边界绑定：`ENV / VERSION / UPDATE / COMPAT`。

## 5. 验证（Verification）

- `V1` 构建注入有效性：开发、测试、打包与发布流程必须可编排执行，且不同阶段环境变量与配置必须被正确注入与隔离。
- `V2` 发布治理有效性：安装包与资源产物必须可识别、可校验、可追踪，版本号与发布顺序必须被正确治理。
- `V3` 更新路径有效性：更新检测、下载、安装与失败回退必须可执行。
- `V4` 兼容符合性：不同平台构建与发布规则必须满足 `COMPAT` 约束。
- `V5` 最小必要性：移除任一 `B1~B2` 后，`V2~V4` 至少一项失败。
- `V6` 结论表达：逐条输出 `R* -> C* / 边界` 的通过或失败结论。
