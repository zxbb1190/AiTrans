# 桌面截图翻译框架:DesktopScreenshotTranslationFramework

@framework

本模块作为 `aitrans` 根层，只汇总 `L2` 领域组合结果，并要求 `L0 / L1 / L2` 的结构调整最终回收到本层统一表达，避免叶子与控制面变化脱离总框架语义。

## 1. 能力声明（Capability Statement）

* `C1` 桌面交互承载能力：提供系统级托盘驻留、全局快捷键、窗口调度与权限协同能力。
* `C2` 截图采集能力：提供桌面区域选择、屏幕图像捕获、图像裁剪与多屏适配能力。
* `C3` 文本识别能力：提供截图内容文字提取、语言初判与识别结果标准化能力。
* `C4` 翻译转换能力：提供源文本翻译、结果格式化与多翻译引擎适配能力。
* `C5` 结果展示能力：提供悬浮展示、原文译文对照、复制与交互反馈能力。
* `C6` 运行治理能力：提供安全控制、日志观测、异常恢复、配置管理与兼容治理能力。
* `C7` 非能力项：不负责具体业务知识库增强、复杂文档排版还原与人工专业术语审校。

## 2. 边界定义（Boundary / 参数）

* `DESKTOP` 桌面宿主约束：必须运行于桌面操作系统，具备托盘、快捷键与窗口管理能力。来源：`C1`。
* `CAPTURE` 截图约束：需支持区域截图、全屏截图与多显示器场景下的坐标映射。来源：`C2`。
* `OCR` 识别约束：识别引擎需支持中英文基础场景，并允许本地或云端切换。来源：`C3`。
* `TRANS` 翻译约束：翻译服务需具备可替换接口、失败回退与响应超时控制。来源：`C4`。
* `UIX` 展示交互约束：结果展示需至少支持关闭、复制与再次触发三类动作，且不得阻塞下一次截图。来源：`C5`。
* `RUNTIME` 运行时约束：需明确主进程、渲染进程与预加载桥接边界，以及允许暴露给渲染进程的 IPC 接口集合。来源：`C1 + C6`。
* `BUILD` 构建发布约束：需明确开发态、打包态、自动更新与平台分发模式。来源：`C1 + C6`。
* `SEC` 安全约束：需限制渲染进程权限、控制 IPC 暴露面、治理第三方依赖与敏感配置。来源：`C1 + C6`。
* `OBS` 观测约束：需覆盖 `capture_started / capture_completed / ocr_requested / trans_requested / result_rendered / failure_raised` 六类关键事件及对应错误指标。来源：`C6`。
* `COMPAT` 兼容约束：需显式声明 Windows / macOS 平台集合、最低支持范围及权限降级策略。来源：`C1 + C6`。
* `PERF` 性能约束：需显式声明截图响应、OCR 阶段、翻译阶段与首屏展示的 `p95` 时延预算。来源：`C2 + C3 + C4 + C5`。

## 3. 最小可行基（Minimum Viable Bases）

* `B1` 桌面触发采集基：L2.M0[桌面宿主层] + L2.M1[截图交互层]。来源：`C1 + C2 + DESKTOP + CAPTURE + RUNTIME + COMPAT + PERF`。
* `B2` 识别翻译闭环基：L2.M2[识别翻译层]。来源：`C3 + C4 + OCR + TRANS + PERF`。
* `B3` 结果交付基：L2.M3[结果展示层]。来源：`C5 + UIX + PERF`。
* `B4` 治理运行基：L2.M4[运行治理层]。来源：`C6 + SEC + OBS + BUILD + RUNTIME + COMPAT`。

## 4. 基组合原则（Base Combination Principles）

* `R1` 桌面触发采集组合

  * `R1.1` 参与基：`B1`。
  * `R1.2` 组合方式：先建立桌面宿主、触发入口与截图采集闭环，形成从系统触发到标准截图结果的统一承接面。
  * `R1.3` 输出能力：`C1 + C2`。
  * `R1.4` 边界绑定：`DESKTOP / CAPTURE / RUNTIME / PERF / COMPAT`。

* `R2` 识别翻译组合

  * `R2.1` 参与基：`B1 + B2`。
  * `R2.2` 组合方式：在桌面触发采集闭环上接入识别翻译主链，形成从截图结果到统一译文契约的处理闭环。
  * `R2.3` 输出能力：`C1 + C2 + C3 + C4`。
  * `R2.4` 边界绑定：`OCR / TRANS / PERF / RUNTIME / SEC`。

* `R3` 结果交付组合

  * `R3.1` 参与基：`B1 + B2 + B3`。
  * `R3.2` 组合方式：将识别翻译结果绑定到低打扰展示与交互结构，形成从系统触发到用户可见交付的完整结果回路。
  * `R3.3` 输出能力：`C1 + C2 + C3 + C4 + C5`。
  * `R3.4` 边界绑定：`UIX / CAPTURE / OCR / TRANS / PERF`。

* `R4` 治理可发布组合

  * `R4.1` 参与基：`B1 + B2 + B3 + B4`。
  * `R4.2` 组合方式：将通信、安全、配置、观测与构建发布规则绑定到完整执行链路，形成可运行、可维护、可发布框架。
  * `R4.3` 输出能力：`C1 + C2 + C3 + C4 + C5 + C6`。
  * `R4.4` 边界绑定：`SEC / OBS / BUILD / RUNTIME / COMPAT / PERF`。

* `R5` 禁止组合

  * `R5.1` 参与基：`B1 + B2 + B3 + B4`。
  * `R5.2` 组合方式：跳过桌面触发采集闭环直接拼接识别结果，或绕过治理边界直接向渲染上下文暴露系统能力的方案均无效。
  * `R5.3` 输出能力：`C7`。
  * `R5.4` 边界绑定：`DESKTOP / CAPTURE / RUNTIME / SEC / OBS / COMPAT`。

## 5. 验证（Verification）

* `V1` 桌面触发采集有效性：桌面入口、截图选区、图像采集与标准截图结果必须形成全链路可执行路径。
* `V2` 识别翻译有效性：OCR 与翻译链路必须在成功、失败、超时三类场景下具备确定性结果或回退结果。
* `V3` 结果交付有效性：结果展示必须满足 `UIX` 约束，支持关闭、复制与再次触发操作。
* `V4` 治理有效性：安全规则、日志事件、错误指标、配置策略与发布约束必须覆盖截图、识别、翻译、展示关键路径。
* `V5` 边界符合性：所有有效组合必须满足 `RUNTIME / OBS / COMPAT / PERF` 约束。
* `V6` 最小必要性：移除任一 `B1~B4` 后，`V2~V5` 至少一项失败。
* `V7` 结论表达：逐条输出 `R* -> C* / 边界` 的通过或失败结论。
