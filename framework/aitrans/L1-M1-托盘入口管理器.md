# 托盘入口管理器:TuoPanRuKouGuanLiQi

@framework

## 1. 能力声明（Capability Statement）

- `C1` 托盘驻留能力：提供应用托盘图标创建、驻留与销毁能力。
- `C2` 菜单装配能力：提供托盘菜单构建、菜单项绑定与动态更新能力。
- `C3` 状态映射能力：提供托盘状态与应用状态同步能力。
- `C4` 快捷入口能力：提供从托盘到截图、设置、退出等核心操作的入口能力。
- `C5` 交互反馈能力：提供点击、双击、右键与状态提示反馈能力。
- `C6` 非能力项：不负责全局快捷键注册与窗口创建实现细节。

## 2. 边界定义（Boundary / 参数）

- `TRAY` 托盘驻留约束：托盘必须在应用运行期间可见且可操作。来源：`C1`。
- `MENU` 菜单约束：菜单项必须具备唯一标识、动作绑定与动态可更新能力。来源：`C2`。
- `STATE` 状态约束：托盘状态必须与应用主状态保持一致。来源：`C3`。
- `ENTRY` 入口约束：托盘必须能映射到核心系统操作入口。来源：`C4`。
- `FEEDBACK` 反馈约束：用户交互后必须具备明确反馈。来源：`C5`。
- `COMPAT` 兼容约束：托盘行为需适配不同平台桌面语义。来源：`C1 + C5`。

## 3. 最小可行基（Minimum Viable Bases）

- `B1` 托盘承载骨架：L0.M10[窗口实例承载器]。来源：`C1 + C2 + TRAY + MENU`。
- `B2` 状态路由骨架：L0.M2[状态机内核] + L0.M3[事件总线]。来源：`C3 + C4 + STATE + ENTRY`。
- `B3` 反馈审计骨架：L0.M9[审计追踪器]。来源：`C5 + FEEDBACK + COMPAT`。

## 4. 基组合原则（Base Combination Principles）

- `R1` 托盘承载组合
  - `R1.1` 参与基：`B1`。
  - `R1.2` 组合方式：先建立托盘实例、驻留状态与菜单承载结构，形成稳定入口壳。
  - `R1.3` 输出能力：`C1 + C2`。
  - `R1.4` 边界绑定：`TRAY / MENU`。

- `R2` 状态路由组合
  - `R2.1` 参与基：`B1 + B2`。
  - `R2.2` 组合方式：将托盘承载结构与状态同步、入口路由绑定，形成可操作且与应用主状态一致的托盘入口。
  - `R2.3` 输出能力：`C1 + C2 + C3 + C4`。
  - `R2.4` 边界绑定：`STATE / ENTRY / MENU`。

- `R3` 反馈审计组合
  - `R3.1` 参与基：`B1 + B2 + B3`。
  - `R3.2` 组合方式：将点击、双击、右键等交互反馈与入口动作追踪绑定到托盘入口闭环。
  - `R3.3` 输出能力：`C1 + C2 + C3 + C4 + C5`。
  - `R3.4` 边界绑定：`FEEDBACK / COMPAT / STATE / ENTRY`。

- `R4` 禁止组合
  - `R4.1` 参与基：`B1 + B2 + B3`。
  - `R4.2` 组合方式：无托盘承载结构直接注入菜单，或入口动作无状态同步与反馈审计的方案均无效。
  - `R4.3` 输出能力：`C6`。
  - `R4.4` 边界绑定：`TRAY / MENU / STATE / FEEDBACK`。

## 5. 验证（Verification）

- `V1` 托盘承载有效性：托盘图标、菜单承载与驻留生命周期必须可执行。
- `V2` 状态路由有效性：托盘状态与应用状态必须同步正确，截图、设置、退出等入口必须可触发。
- `V3` 反馈审计有效性：交互动作必须具备明确反馈并可追踪。
- `V4` 兼容符合性：托盘行为必须满足 `COMPAT` 约束。
- `V5` 最小必要性：移除任一 `B1~B3` 后，`V2~V4` 至少一项失败。
- `V6` 结论表达：逐条输出 `R* -> C* / 边界` 的通过或失败结论。
