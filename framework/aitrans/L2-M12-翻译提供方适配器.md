# 模块框架：L2-M12-翻译提供方适配器

@module

## 1. 能力声明（Capability Statement）

- `C1` 统一接入能力：提供翻译 Provider 的统一接口接入能力。
- `C2` 请求构造能力：提供原文、源语言、目标语言与附加参数的请求构造能力。
- `C3` 结果解析能力：提供译文提取、格式清洗与补充元信息整理能力。
- `C4` 错误分流能力：提供翻译失败、响应异常与空译文分流能力。
- `C5` 多引擎兼容能力：提供多翻译引擎接入下的统一兼容能力。
- `C6` 非能力项：不负责 OCR 图像识别与展示渲染。

## 2. 边界定义（Boundary / 参数）

- `PROVIDER` 接口约束：翻译引擎必须经统一 Provider 接口暴露。来源：`C1`。
- `REQUEST` 请求约束：原文、语言对与策略参数必须统一建模。来源：`C2`。
- `PARSE` 解析约束：翻译响应必须可提取为标准译文与元信息。来源：`C3`。
- `ERROR` 错误约束：错误必须具备分类、封装与回传能力。来源：`C4`。
- `COMPAT` 兼容约束：不同翻译引擎差异必须在适配层被收敛。来源：`C5`。
- `SEC` 安全约束：翻译调用中密钥、文本与服务配置应受控访问。来源：`C2 + C4 + C5`。

## 3. 最小可行基（Minimum Viable Bases）

- `B1` Provider 接口基：由 Translator Provider 接口、能力描述器与实现绑定器构成。来源：`C1 + PROVIDER`。
- `B2` 请求构造基：framework.L3.M5[请求构造器]。来源：`C2 + REQUEST`。
- `B3` 结果解析基：framework.L3.M6[结果解析器]。来源：`C3 + PARSE`。
- `B4` 错误分流基：framework.L3.M7[错误分流器]。来源：`C4 + ERROR`。
- `B5` 兼容映射基：由能力映射器、字段映射器、差异收敛器与引擎选择器构成。来源：`C5 + COMPAT + SEC`。

## 4. 基组合原则（Base Combination Principles）

- `R1` 接口先行组合
  - `R1.1` 参与基：`B1`。
  - `R1.2` 组合方式：先建立统一翻译 Provider 接口，再接入请求与解析链路。
  - `R1.3` 输出能力：`C1`。
  - `R1.4` 边界绑定：`PROVIDER`。

- `R2` 请求解析组合
  - `R2.1` 参与基：`B1 + B2 + B3`。
  - `R2.2` 组合方式：在统一接口之上绑定翻译请求构造与结果解析，形成标准翻译调用链路。
  - `R2.3` 输出能力：`C1 + C2 + C3`。
  - `R2.4` 边界绑定：`REQUEST / PARSE`。

- `R3` 错误保护组合
  - `R3.1` 参与基：`B2 + B3 + B4`。
  - `R3.2` 组合方式：在请求与解析链路上叠加翻译失败与异常分流能力。
  - `R3.3` 输出能力：`C2 + C3 + C4`。
  - `R3.4` 边界绑定：`ERROR / SEC`。

- `R4` 多引擎兼容组合
  - `R4.1` 参与基：`B1 + B3 + B5`。
  - `R4.2` 组合方式：通过兼容映射层收敛多翻译引擎差异，形成统一译文输出语义。
  - `R4.3` 输出能力：`C1 + C3 + C5`。
  - `R4.4` 边界绑定：`COMPAT / PROVIDER`。

- `R5` 完整翻译适配模块组合
  - `R5.1` 参与基：`B1 + B2 + B3 + B4 + B5`。
  - `R5.2` 组合方式：将统一接入、请求构造、结果解析、错误分流与兼容映射整合为完整翻译适配模块。
  - `R5.3` 输出能力：`C1 + C2 + C3 + C4 + C5`。
  - `R5.4` 边界绑定：`PROVIDER / REQUEST / PARSE / ERROR / COMPAT / SEC`。

- `R6` 禁止组合
  - `R6.1` 参与基：`B1 + B2 + B3 + B4 + B5`。
  - `R6.2` 组合方式：绕过统一接口直接耦合具体翻译 SDK，或不做错误封装直接向上抛出原始异常的方案均无效。
  - `R6.3` 输出能力：`C6`。
  - `R6.4` 边界绑定：`PROVIDER / ERROR / COMPAT`。

## 5. 验证（Verification）

- `V1` 接入有效性：翻译 Provider 必须通过统一接口接入。
- `V2` 请求有效性：原文、语言参数与策略参数必须正确构造请求。
- `V3` 解析有效性：译文与元信息必须被正确提取与清洗。
- `V4` 错误有效性：失败、异常与空译文必须被正确分流与回传。
- `V5` 兼容有效性：多翻译引擎差异必须在适配层被收敛。
- `V6` 安全符合性：文本与密钥访问必须满足 `SEC` 约束。
- `V7` 最小必要性：移除任一 `B1~B5` 后，`V2~V6` 至少一项失败。
- `V8` 结论表达：逐条输出 `R* -> C* / 边界` 的通过或失败结论。
