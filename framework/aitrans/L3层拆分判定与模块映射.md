# framework L3 层拆分判定与模块映射

## 1. 判定规则

- `继续下钻到 L3`：同一 `L2` 模块同时具备明显状态机、阶段编排、策略决策面、多个稳定子接口中的至少 `3` 项，并且还能自然拆成 `4` 个以上非工具型职责块。
- `仅吸收到共享 L3`：`L2` 模块本身不值得继续裂成大量专用块，但其内部存在稳定共性骨架，例如会话初始化、上下文存储、请求构造、结果解析、错误分流、审计追踪。
- `停在 L2`：继续拆只会变成 `Handler / Builder / Mapper / Validator / UI 子组件 / 适配细节`，对架构决策帮助很小。
- `L3 停止继续下钻`：当子模块已经变成单一协议、单一状态职责或单一控制面时，后续应转入接口设计与实现设计，不再继续写 `L4` 框架文档。

| --- | --- | --- | --- |
| `L2-M0` 应用生命周期管理器 | 停在 `L2` | 再拆会退化为启动器、守卫、清理器实现类 | 可选复用 `L3-M3` |
| `L2-M1` 托盘入口管理器 | 停在 `L2` | 已接近平台入口适配器与菜单实现对象 | 不新增 |
| `L2-M2` 全局快捷键路由器 | 停在 `L2` | 再拆将落入注册器、冲突处理器与路由器实现细节 | 可选复用 `L3-M9` |
| `L2-M3` 窗口调度器 | 下钻到 `L3` | 同时具备工厂、注册、显隐、销毁、多窗协同五类控制面 | `L3-M10 ~ L3-M14` |
| `L2-M4` 权限协调器 | 停在 `L2` | 再拆后多数是探测器、缓存器、引导器、通知器实现对象 | 不新增 |
| `L2-M5` 截图会话管理器 | 共享吸收到 `L3` | 核心骨架可被会话类模块共用，无需再生专用块 | `L3-M0 ~ L3-M4` |
| `L2-M6` 遮罩选区控制器 | 下钻到 `L3` | 输入、选区模型、视觉反馈、确认取消边界清晰 | `L3-M15 ~ L3-M19` |
| `L2-M7` 屏幕帧获取器 | 停在 `L2` | 再拆更像源枚举、裁剪、编码实现链路 | 不新增 |
| `L2-M8` 坐标映射引擎 | 下钻到 `L3` | 显示器拓扑、缩放、转换、多屏归属、边界修正可稳定分离 | `L3-M20 ~ L3-M24` |
| `L2-M9` 截图结果标准化器 | 停在 `L2` | 更像标准化器与适配器组合，再拆价值有限 | 不新增 |

| `L2-M11` 文字识别提供方适配器 | 共享吸收到 `L3` | 重复点集中在请求构造、结果解析、错误分流 | `L3-M5 ~ L3-M7` |
| `L2-M12` 翻译提供方适配器 | 共享吸收到 `L3` | 与 `L2-M11` 镜像，适合直接复用共享骨架 | `L3-M5 ~ L3-M7` |
| `L2-M13` 回退策略控制器 | 下钻到 `L3` | 失败分类、超时、重试、切换、降级构造均为独立策略点 | `L3-M27 ~ L3-M31` |
| `L2-M14` 识别翻译结果装配器 | 共享吸收到 `L3` | 核心重叠在错误收敛与标准输出协议 | `L3-M7 ~ L3-M8` |
| `L2-M15` 展示会话管理器 | 共享吸收到 `L3` | 与截图会话同构，复用共享会话骨架即可 | `L3-M0 ~ L3-M4` |
| `L2-M16` 结果视图渲染器 | 停在 `L2` | 再拆会滑向 UI 组件树与视图细节 | 不新增 |
| `L2-M17` 交互动作控制器 | 停在 `L2` | 再拆将落入动作 Handler 级别 | 可选复用 `L3-M9` |
| `L2-M18` 悬浮面板布局管理器 | 下钻到 `L3` | 锚点、布局、拖拽、显隐、避障均为稳定控制面 | `L3-M32 ~ L3-M36` |

| `L2-M20` 进程间通信桥接网关 | 下钻到 `L3` | 通道、白名单、校验、响应装配构成明确控制面 | `L3-M9`、`L3-M41 ~ L3-M44` |
| `L2-M21` 配置中心 | 停在 `L2` | 当前项目不是独立配置平台，继续下钻收益不足 | 不新增 |
| `L2-M22` 可观测性中心 | 下钻到 `L3` | 日志、指标、错误、链路、诊断天然是独立观测子面 | `L3-M45 ~ L3-M49` |
| `L2-M23` 安全策略守卫 | 下钻到 `L3` | 权限、访问、依赖、CSP、密钥保护都是独立控制面 | `L3-M50 ~ L3-M54` |
| `L2-M24` 构建发布管理器 | 下钻到 `L3` | 构建、环境、产物、版本、更新可稳定分为五个工程子面 | `L3-M55 ~ L3-M59` |

## 5. L3 停止边界

- 当前 `L3` 的目标是把 `L2` 中仍然存在的“控制面、协议面、状态面、策略面”抽出来，不把 `L3` 继续写成函数清单。
- 因此本轮所有 `L3` 默认停在 `L3`；下一步如果继续细化，应转入接口文档、状态图、时序图和实现约束，而不是继续机械拆成 `L4`。
