# 交互动作控制器:JiaoHuDongZuoKongZhiQi

@framework

## 1. 能力声明（Capability Statement）

- `C1` 动作接入能力：提供复制、关闭、重试、展开与折叠等动作接入能力。
- `C2` 动作分发能力：提供用户动作到具体处理逻辑的分发能力。
- `C3` 反馈提示能力：提供动作成功、失败与处理中反馈提示能力。
- `C4` 节流保护能力：提供高频动作限制、重复触发保护与并发控制能力。
- `C5` 动作审计能力：提供动作触发记录、结果记录与异常追踪能力。
- `C6` 非能力项：不负责结果视图绘制与悬浮面板定位。

## 2. 边界定义（Boundary / 参数）

- `ACTION` 动作约束：交互动作必须具备唯一标识与明确处理目标。来源：`C1`。
- `DISPATCH` 分发约束：动作触发必须被路由到唯一处理器。来源：`C2`。
- `FEEDBACK` 反馈约束：每次动作执行必须具备明确反馈。来源：`C3`。
- `THROTTLE` 节流约束：高频动作必须受限且防止重复执行。来源：`C4`。
- `AUDIT` 审计约束：动作触发过程必须可记录、可追踪。来源：`C5`。
- `UX` 体验约束：动作执行不得阻塞主展示流程。来源：`C2 + C3 + C4`。

## 3. 最小可行基（Minimum Viable Bases）

- `B1` 交互分发骨架：L0.M3[事件总线] + L0.M2[状态机内核]。来源：`C1 + C2 + ACTION + DISPATCH`。
- `B2` 反馈保护骨架：L0.M8[标准输出装配器] + L0.M2[状态机内核] + L0.M9[审计追踪器]。来源：`C3 + C4 + C5 + FEEDBACK + THROTTLE + AUDIT + UX`。

## 4. 基组合原则（Base Combination Principles）

- `R1` 交互分发组合
  - `R1.1` 参与基：`B1`。
  - `R1.2` 组合方式：先定义动作标识、状态驱动与处理器分发关系，形成统一交互分发面。
  - `R1.3` 输出能力：`C1 + C2`。
  - `R1.4` 边界绑定：`ACTION / DISPATCH`。

- `R2` 反馈保护组合
  - `R2.1` 参与基：`B1 + B2`。
  - `R2.2` 组合方式：在交互分发链路上叠加反馈提示、节流去重与审计追踪，形成可感知且可保护的动作闭环。
  - `R2.3` 输出能力：`C1 + C2 + C3 + C4 + C5`。
  - `R2.4` 边界绑定：`FEEDBACK / THROTTLE / AUDIT / UX`。

- `R3` 禁止组合
  - `R3.1` 参与基：`B1 + B2`。
  - `R3.2` 组合方式：无交互分发骨架直接处理动作，或无反馈保护骨架直接消费高频触发的方案均无效。
  - `R3.3` 输出能力：`C6`。
  - `R3.4` 边界绑定：`ACTION / THROTTLE / AUDIT`。

## 5. 验证（Verification）

- `V1` 交互分发有效性：交互动作必须可注册并路由到唯一处理器执行。
- `V2` 反馈保护有效性：动作执行中、成功与失败提示必须可触发，高频动作必须受到节流与重复执行保护。
- `V3` 审计与体验有效性：动作触发、结果与异常必须可追踪，且动作执行不得破坏 `UX` 约束。
- `V4` 最小必要性：移除任一 `B1~B2` 后，`V1~V3` 至少一项失败。
- `V5` 结论表达：逐条输出 `R* -> C* / 边界` 的通过或失败结论。
