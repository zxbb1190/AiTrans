# 模块框架：L1-M4-运行治理层

@module

## 1. 能力声明（Capability Statement）

- `C1` 进程桥接治理能力：提供主进程、预加载与渲染进程之间的受控通信能力。
- `C2` 配置管理能力：提供本地配置、敏感参数、环境差异与运行时读取能力。
- `C3` 日志观测能力：提供关键链路日志、错误采集、指标记录与问题定位能力。
- `C4` 安全控制能力：提供 IPC 暴露治理、依赖约束、内容安全与敏感数据隔离能力。
- `C5` 构建发布治理能力：提供开发态、打包态、发布态与版本更新治理能力。
- `C6` 非能力项：不负责业务截图交互、OCR 识别与结果展示本身。

## 2. 边界定义（Boundary / 参数）

- `IPC` 通信约束：进程通信必须通过白名单接口、请求校验与响应约束进行控制。来源：`C1`。
- `CONF` 配置约束：配置必须区分公开参数、敏感参数与环境参数。来源：`C2`。
- `OBS` 观测约束：日志、指标与错误上报必须覆盖关键执行链路。来源：`C3`。
- `SEC` 安全约束：必须限制渲染进程能力暴露、依赖风险与不受控资源访问。来源：`C4`。
- `BUILD` 构建约束：必须明确开发、测试、打包、发布与更新链路。来源：`C5`。
- `COMPAT` 兼容约束：不同平台与不同分发形态下治理规则需保持一致。来源：`C4 + C5`。
- `AUDIT` 审计约束：关键操作必须可追踪、可回放定位。来源：`C3 + C4`。

## 3. 最小可行基（Minimum Viable Bases）

- `B1` IPC 治理基：framework.L2.M20[进程间通信桥接网关]。来源：`C1 + IPC + SEC`。
- `B2` 配置中心基：framework.L2.M21[配置中心]。来源：`C2 + CONF`。
- `B3` 观测日志基：framework.L2.M22[可观测性中心]。来源：`C3 + OBS + AUDIT`。
- `B4` 安全控制基：framework.L2.M23[安全策略守卫]。来源：`C4 + SEC + COMPAT`。
- `B5` 构建发布基：framework.L2.M24[构建发布管理器]。来源：`C5 + BUILD + COMPAT`。

## 4. 基组合原则（Base Combination Principles）

- `R1` 通信先行组合
  - `R1.1` 参与基：`B1`。
  - `R1.2` 组合方式：先建立进程桥接白名单与校验规则，再允许上层模块接入通信链路。
  - `R1.3` 输出能力：`C1`。
  - `R1.4` 边界绑定：`IPC / SEC`。

- `R2` 配置观测组合
  - `R2.1` 参与基：`B2 + B3`。
  - `R2.2` 组合方式：将环境配置与日志观测绑定，形成可配置、可定位的运行治理能力。
  - `R2.3` 输出能力：`C2 + C3`。
  - `R2.4` 边界绑定：`CONF / OBS / AUDIT`。

- `R3` 安全闭环组合
  - `R3.1` 参与基：`B1 + B4`。
  - `R3.2` 组合方式：将 IPC 治理与安全控制策略绑定，形成从接口暴露到运行访问的安全闭环。
  - `R3.3` 输出能力：`C1 + C4`。
  - `R3.4` 边界绑定：`IPC / SEC / COMPAT`。

- `R4` 构建发布组合
  - `R4.1` 参与基：`B2 + B4 + B5`。
  - `R4.2` 组合方式：将配置、安全与打包发布策略绑定，形成面向多环境和多平台的发布治理链路。
  - `R4.3` 输出能力：`C2 + C4 + C5`。
  - `R4.4` 边界绑定：`CONF / SEC / BUILD / COMPAT`。

- `R5` 完整治理模块组合
  - `R5.1` 参与基：`B1 + B2 + B3 + B4 + B5`。
  - `R5.2` 组合方式：将进程桥接、配置管理、日志观测、安全控制与构建发布整合为完整治理运行模块。
  - `R5.3` 输出能力：`C1 + C2 + C3 + C4 + C5`。
  - `R5.4` 边界绑定：`IPC / CONF / OBS / SEC / BUILD / COMPAT / AUDIT`。

- `R6` 禁止组合
  - `R6.1` 参与基：`B1 + B2 + B3 + B4 + B5`。
  - `R6.2` 组合方式：绕过 IPC 白名单直接暴露系统能力，或缺少日志观测与安全控制的发布方案均无效。
  - `R6.3` 输出能力：`C6`。
  - `R6.4` 边界绑定：`IPC / OBS / SEC / BUILD / AUDIT`。

## 5. 验证（Verification）

- `V1` 通信治理有效性：进程桥接接口必须受白名单、校验与错误映射控制。
- `V2` 配置有效性：配置加载、环境区分与敏感参数访问必须满足 `CONF` 约束。
- `V3` 观测有效性：日志、指标与错误采集必须覆盖关键执行链路。
- `V4` 安全有效性：权限最小化、依赖治理与内容安全策略必须可执行。
- `V5` 发布有效性：构建、打包、版本与更新链路必须满足 `BUILD` 约束。
- `V6` 边界符合性：所有有效组合必须满足 `COMPAT / AUDIT` 约束。
- `V7` 最小必要性：移除任一 `B1~B5` 后，`V2~V6` 至少一项失败。
- `V8` 结论表达：逐条输出 `R* -> C* / 边界` 的通过或失败结论。
