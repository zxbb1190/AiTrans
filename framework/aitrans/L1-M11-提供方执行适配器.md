# 提供方执行适配器:TiGongFangZhiXingShiPeiQi

@framework

## 1. 能力声明（Capability Statement）

- `C1` 接入承接能力：提供外部 Provider 的统一接入边界与实现挂载能力。
- `C2` 请求封装能力：提供图像、文本、语言提示与策略参数的统一请求封装能力。
- `C3` 响应解释能力：提供执行结果提取、字段清洗与执行元信息归一能力。
- `C4` 失败收敛能力：提供调用失败、异常响应与空结果的统一收敛能力。
- `C5` 差异收敛能力：提供多 Provider 差异的适配层内收敛能力。
- `C6` 非能力项：不负责任务编排、回退策略决策与结果展示。

## 2. 边界定义（Boundary / 参数）

- `PROVIDER` 接口约束：OCR 与翻译 Provider 必须经统一接入边界暴露，且不得向上游暴露原生 SDK 形态。来源：`C1`。
- `REQUEST` 请求约束：图像、文本、语言对、区域参数与执行策略必须统一建模。来源：`C2`。
- `RESPONSE` 响应约束：响应必须可归一为文本结果、语言信息与执行元信息。来源：`C3`。
- `FAILURE` 失败约束：失败必须可区分网络、鉴权、限流、服务异常与空结果等类型。来源：`C4`。
- `COMPAT` 兼容约束：Provider 差异只能停留在请求映射、响应解释与失败归一边界。来源：`C5`。
- `SEC` 安全约束：图像、文本与敏感配置只能在适配上下文内最小暴露。来源：`C2 + C4 + C5`。

## 3. 最小可行基（Minimum Viable Bases）

- `B1` 接入请求壳基：L0.M5[请求构造器]。来源：`C1 + C2 + PROVIDER + REQUEST`。
- `B2` 响应解释壳基：L0.M6[结果解析器]。来源：`C3 + RESPONSE`。
- `B3` 失败收敛壳基：L0.M7[错误分流器]。来源：`C4 + C5 + FAILURE + COMPAT + SEC`。

## 4. 基组合原则（Base Combination Principles）

- `R1` 接入请求组合
  - `R1.1` 参与基：`B1`。
  - `R1.2` 组合方式：先承接统一 Provider 接口，再完成图像、文本与执行参数的统一请求封装。
  - `R1.3` 输出能力：`C1 + C2`。
  - `R1.4` 边界绑定：`PROVIDER / REQUEST`。

- `R2` 响应解释组合
  - `R2.1` 参与基：`B1 + B2`。
  - `R2.2` 组合方式：在统一请求壳之上绑定响应解释边界，形成可归一的 Provider 调用链路。
  - `R2.3` 输出能力：`C1 + C2 + C3`。
  - `R2.4` 边界绑定：`REQUEST / RESPONSE`。

- `R3` 失败兼容组合
  - `R3.1` 参与基：`B1 + B2 + B3`。
  - `R3.2` 组合方式：将失败分类、异常收敛与多 Provider 差异收敛叠加到执行链路，形成稳定执行适配面。
  - `R3.3` 输出能力：`C1 + C2 + C3 + C4 + C5`。
  - `R3.4` 边界绑定：`FAILURE / COMPAT / SEC`。

- `R4` 禁止组合
  - `R4.1` 参与基：`B1 + B2 + B3`。
  - `R4.2` 组合方式：绕过统一接入边界直接耦合具体 Provider SDK，或向上游暴露原始响应与原始异常的方案均无效。
  - `R4.3` 输出能力：`C6`。
  - `R4.4` 边界绑定：`PROVIDER / FAILURE / COMPAT`。

## 5. 验证（Verification）

- `V1` 接入有效性：OCR 与翻译 Provider 必须均经统一接入边界挂载。
- `V2` 请求有效性：图像、文本、语言提示与执行参数必须被统一封装。
- `V3` 响应有效性：执行结果必须被归一为文本结果与执行元信息。
- `V4` 失败有效性：失败、异常与空结果必须被正确分类并完成统一收敛。
- `V5` 兼容有效性：多 Provider 差异必须停留在适配边界内。
- `V6` 安全符合性：输入内容与敏感配置访问必须满足 `SEC` 约束。
- `V7` 最小必要性：移除任一 `B1~B3` 后，`V2~V6` 至少一项失败。
- `V8` 结论表达：逐条输出 `R* -> C* / 边界` 的通过或失败结论。
