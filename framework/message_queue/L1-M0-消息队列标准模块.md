# 消息队列标准模块:MessageQueueStandardModule

@framework

## 1. 能力声明（Capability Statement）

- `C1` 生产接入编排能力：将消息生产入口、路由选择与入队条件编排为稳定接入主链。
- `C2` 消费确认编排能力：将订阅、消费、确认与重投递条件编排为稳定消费闭环。
- `C3` 积压失败治理能力：将积压观测、失败隔离与回收路径编排为可维护治理链。
- `C4` 非能力项：不负责具体消息中间件产品、SDK 调用细节和业务处理逻辑。

## 2. 边界定义（Boundary / 参数）

- `INGRESS` 接入边界：生产者标识、发送批量、幂等键与入队条件需明确。来源：`C1 + C2`。
- `SUBSCRIBE` 订阅边界：消费者组、并发窗口、分配范围与重平衡粒度需可区分。来源：`C1 + C2`。
- `ACK` 确认边界：确认时限、拒收语义、重投递触发与检查点需可验证。来源：`C2 + C3`。
- `BACKLOG` 积压边界：积压阈值、滞留时长与回收节奏需可量化。来源：`C2 + C3`。
- `FAILURE` 失败边界：失败分类、死信入口、回放条件与隔离出口需可执行。来源：`C2 + C3`。
- `ORDER` 顺序边界：生产和消费两侧的顺序保持粒度需可追踪。来源：`C1 + C2`。
- `TRACE` 追踪边界：入队、投递、确认、重试与失败事件需可关联。来源：`C1 + C3`。
- `SLA` 服务边界：吞吐、延迟与可恢复窗口需可判定。来源：`C1 + C3`。

## 3. 最小可行基（Minimum Viable Bases）

- `B1` 序列拓扑接入编排基：L0.M0[R1,R2]。来源：`C1 + INGRESS + ORDER + SLA`。
- `B2` 拓扑游标消费编排基：L0.M0[R2,R3]。来源：`C2 + SUBSCRIBE + ACK + ORDER`。
- `B3` 游标失败治理编排基：L0.M0[R3]。来源：`C3 + BACKLOG + FAILURE + TRACE`。

## 4. 基组合原则（Base Combination Principles）

- `R1` 接入主链组合
  - `R1.1` 参与基：`B1`。
  - `R1.2` 组合方式：先固定生产入口、路由粒度与入队条件，再开放消费侧编排。
  - `R1.3` 输出能力：`C1`。
  - `R1.4` 边界绑定：`INGRESS + ORDER + SLA`。
- `R2` 消费确认组合
  - `R2.1` 参与基：`B1 + B2`。
  - `R2.2` 组合方式：将订阅关系、消费分配与确认机制绑定到接入主链，形成稳定消费闭环。
  - `R2.3` 输出能力：`C1 + C2`。
  - `R2.4` 边界绑定：`SUBSCRIBE + ACK + ORDER + INGRESS`。
- `R3` 治理闭环组合
  - `R3.1` 参与基：`B1 + B2 + B3`。
  - `R3.2` 组合方式：把积压观测、失败隔离与回放路径接入消费闭环，形成端到端治理系统。
  - `R3.3` 输出能力：`C1 + C2 + C3`。
  - `R3.4` 边界绑定：`BACKLOG + FAILURE + TRACE + ACK + SLA`。
- `R4` 禁止组合
  - `R4.1` 参与基：`B1 + B2 + B3`。
  - `R4.2` 组合方式：缺失接入主链、确认语义不可判定或失败消息无隔离回收路径的组合无效。
  - `R4.3` 输出能力：`C4`。
  - `R4.4` 边界绑定：`INGRESS + SUBSCRIBE + ACK + BACKLOG + FAILURE + ORDER + TRACE + SLA`。

## 5. 验证（Verification）

- `V1` 接入一致性：生产接入主链必须满足 `INGRESS + ORDER + SLA` 约束。
- `V2` 消费有效性：订阅、消费与确认链必须满足 `SUBSCRIBE + ACK + ORDER` 约束。
- `V3` 治理有效性：积压观测、失败隔离与回放路径必须满足 `BACKLOG + FAILURE + TRACE` 约束。
- `V4` 最小必要性：移除任一 `B*` 后，`V2~V3` 至少一项失败。
- `V5` 结论表达：逐条输出 `R* -> C* + 边界绑定` 的通过或失败结论。
