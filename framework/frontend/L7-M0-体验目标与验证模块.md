# 体验目标验证模块:ExperienceGoalVerification

@framework

## 1. 能力声明（Capability Statement）

- `C1` 体验目标建模能力：将体验目标转为可量化指标体系。
- `C2` 多维验证能力：支持性能、稳定性、可用性和一致性联合验证。
- `C3` 反馈优化能力：支持异常告警、回归识别和持续优化闭环。
- `C4` 非能力项：不负责底层组件实现与业务规则定义。

## 2. 边界定义（Boundary / 参数）

- `SPEED` 速度指标：首屏、交互和流程耗时阈值需明确。来源：`C1 + C2`。
- `STABLE` 稳定性指标：错误率、崩溃率和重试率需可观测。来源：`C2 + C3`。
- `A11Y` 可访问指标：键盘可达与语义可读指标需达标。来源：`C1 + C2`。
- `CONSIST` 一致性指标：同类交互反馈和视觉语义需一致。来源：`C1 + C2`。
- `FUNNEL` 流程漏斗指标：登录等关键流程转化节点需可追踪。来源：`C1 + C3`。
- `SAT` 用户满意代理指标：任务完成率与反馈评分需可量化。来源：`C1 + C3`。
- `ALERT` 告警策略：指标越界触发条件和升级路径需明确。来源：`C2 + C3`。

## 3. 最小可行基（Minimum Viable Bases）

- `B1` 指标模型基：L6.M0[R1,R2,R3]。来源：`C1 + SPEED + STABLE + CONSIST`。
- `B2` 验证编排基：L6.M0[R2]。来源：`C1 + C2 + FUNNEL + A11Y + SPEED`。
- `B3` 告警回归基：L6.M0[R2,R3]。来源：`C2 + C3 + ALERT + SAT + STABLE`。

## 4. 基组合原则（Base Combination Principles）

- `R1` 指标建模组合
  - `R1.1` 参与基：`B1`。
  - `R1.2` 组合方式：先统一指标口径和采样边界，再接入验证执行。
  - `R1.3` 输出能力：`C1`。
  - `R1.4` 边界绑定：`SPEED/STABLE/CONSIST`。
- `R2` 验证联动组合
  - `R2.1` 参与基：`B1 + B2`。
  - `R2.2` 组合方式：以统一指标驱动自动化验证和流程漏斗跟踪。
  - `R2.3` 输出能力：`C1 + C2`。
  - `R2.4` 边界绑定：`FUNNEL/A11Y/SPEED/CONSIST`。
- `R3` 反馈闭环组合
  - `R3.1` 参与基：`B1 + B2 + B3`。
  - `R3.2` 组合方式：将验证结果接入告警与回归治理，形成持续优化闭环。
  - `R3.3` 输出能力：`C1 + C2 + C3`。
  - `R3.4` 边界绑定：`ALERT/SAT/STABLE/FUNNEL`。
- `R4` 禁止组合
  - `R4.1` 参与基：`B1 + B2 + B3`。
  - `R4.2` 组合方式：缺失统一指标口径、不可复现验证路径或无告警回路的方案均无效。
  - `R4.3` 输出能力：`C4`。
  - `R4.4` 边界绑定：`SPEED/STABLE/A11Y/CONSIST/FUNNEL/SAT/ALERT`。

## 5. 验证（Verification）

- `V1` 指标一致性：同一指标在不同环境的口径必须一致。
- `V2` 采样有效性：验证样本必须覆盖关键页面和关键流程节点。
- `V3` 闭环完整性：指标异常必须有告警与回归处置记录。
- `V4` 边界符合性：验证结果必须满足 `SPEED/STABLE/A11Y` 阈值。
- `V5` 最小必要性：移除任一 `B*` 后，`V2~V4` 至少一项失败。
- `V6` 结论表达：逐条输出 `R* -> C* / 边界` 的通过或失败结论。
