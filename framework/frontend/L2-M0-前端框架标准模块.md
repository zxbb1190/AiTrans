# 前端框架标准模块:FrontendFrameworkStandard

@framework

本模块建立在“前端共相结构 -> 基础组件原子 -> 前端标准 -> 更高层前端编排”这条链上。槽位壳体、状态通道、宿主绑定、令牌主题与运行底座位于 `frontend/L0-M0 ~ L0-M4`，按钮、输入框、文本域、文件选择、文本块、面板、列表、反馈与通用装配支撑面位于 `frontend/L1-M0 ~ L1-M8`，本模块只定义前端通用框架如何承接这些结构。项目实例应把本模块编译成显式前端 contract 和规则校验，而不是在实例配置中重写这些结构。对知识工作台这类实例，典型形态是“侧栏 / 主舞台 / 支撑区”三段式承载面。实例配置采用“大类边界 -> 大类 section”约定：`SURFACE -> [surface]`、`VISUAL -> [visual]`、`ROUTE -> [route]`、`A11Y -> [a11y]`；`INTERACT/STATE/EXTEND` 当前仍由框架 contract 固定，不直接开放为实例 section。

## 1. 能力声明（Capability Statement）

- `C1` 通用界面装配能力：把输入、展示、导航与反馈原子装配为稳定前端界面结构。
- `C2` 跨场景交互承接能力：以统一结构承接浏览、编辑、对话、管理等不同场景交互。
- `C3` 领域框架承接能力：为更高层的领域通用框架输出稳定 UI 承接面，而不泄漏底层实现细节。
- `C4` 非能力项：不负责具体领域对象定义、后端存储模型与项目实例值。

## 2. 边界定义（Boundary / 参数）

- `SURFACE` 界面边界：头部、侧栏、主体、支撑区、浮层与反馈区等承载面职责需明确。来源：`C1 + C2`。
- `VISUAL` 视觉边界：颜色、字号级、圆角级、阴影级与密度档位需通过受控 token 与 recipe 表达。来源：`C1 + C3`。
- `INTERACT` 交互边界：输入、选择、切换、展开、创建、删除与提交等交互出口需稳定。来源：`C1 + C2`。
- `STATE` 状态边界：加载、空态、错误与成功等结构状态需统一表达。来源：`C1 + C2`。
- `EXTEND` 扩展边界：领域框架只能在既有插接口上扩展，不得重写通用原子职责。来源：`C2 + C3`。
- `ROUTE` 路由边界：通用页面入口、返回路径与深链落点需可承接。来源：`C2 + C3`。
- `A11Y` 可访问边界：阅读顺序、键盘导航与提示关系需可达。来源：`C1 + C2`。

## 3. 最小可行基（Minimum Viable Bases）

- `B1` 界面装配基：L1.M0[R1,R3] + L1.M1[R1,R3] + L1.M2[R1,R3] + L1.M3[R1,R3] + L1.M8[R1,R2]。来源：`C1 + SURFACE + VISUAL + INTERACT + STATE`。
- `B2` 交互契约基：L1.M4[R1,R3] + L1.M5[R1,R3] + L1.M6[R1,R3] + L1.M7[R1,R3] + L1.M8[R2,R3]。来源：`C1 + C2 + INTERACT + ROUTE + A11Y`。
- `B3` 领域承接基：L1.M0[R2,R3] + L1.M5[R2,R3] + L1.M8[R3]。来源：`C2 + C3 + EXTEND + SURFACE + VISUAL + STATE`。

## 4. 基组合原则（Base Combination Principles）

- `R1` 通用承载面收敛
  - `R1.1` 参与基：`B1`。
  - `R1.2` 组合方式：先固定通用输入、展示、导航和反馈原子的承载面关系，再把 surface 与 visual 大类 section 中的实例值编译成稳定 layout recipe 和 token。
  - `R1.3` 输出能力：`C1`。
  - `R1.4` 边界绑定：`SURFACE/VISUAL/INTERACT/STATE`。
- `R2` 场景交互统一
  - `R2.1` 参与基：`B1 + B2`。
  - `R2.2` 组合方式：在统一承载面上约束交互出口、返回路径与可访问顺序，使不同场景共享同一前端结构骨架，例如侧栏管理、主舞台输入与支撑区回看。
  - `R2.3` 输出能力：`C1 + C2`。
  - `R2.4` 边界绑定：`INTERACT/ROUTE/A11Y/STATE`。
- `R3` 领域扩展承接
  - `R3.1` 参与基：`B1 + B2 + B3`。
  - `R3.2` 组合方式：为领域通用框架提供稳定插接口和扩展槽位，使领域结构建立在通用前端结构之上。
  - `R3.3` 输出能力：`C1 + C2 + C3`。
  - `R3.4` 边界绑定：`EXTEND/SURFACE/VISUAL/STATE/ROUTE/A11Y`。
- `R4` 禁止组合
  - `R4.1` 参与基：`B1 + B2 + B3`。
  - `R4.2` 组合方式：让领域框架直接改写通用原子职责、绕过交互契约、越界声明视觉 recipe 或破坏承载面边界的方案均无效。
  - `R4.3` 输出能力：`C4`。
  - `R4.4` 边界绑定：`SURFACE/VISUAL/INTERACT/STATE/EXTEND/ROUTE/A11Y`。

## 5. 验证（Verification）

- `V1` 装配完整性：通用界面必须同时具备输入、展示、导航和反馈承载面。
- `V2` 交互一致性：不同场景必须共享统一交互出口与返回结构。
- `V3` 扩展稳定性：领域扩展必须通过 `EXTEND` 插接口完成，不得破坏通用职责。
- `V7` 配置映射稳定性：`[surface]/[visual]/[route]/[a11y]` 中的字段必须只落到对应大类边界，不得跨大类游离。
- `V4` 可访问符合性：有效组合必须满足 `A11Y` 约束。
- `V5` 最小必要性：移除任一 `B*` 后，`V2~V4` 至少一项失败。
- `V6` 结论表达：逐条输出 `R* -> C* / 边界` 的通过或失败结论。
