# 前端组合框架模块:FrontendCompositionFramework

@framework

## 1. 能力声明（Capability Statement）

- `C1` 基础组件沉淀能力：将 Button、Input 等原子组件标准化为可复用资产。
- `C2` 页面组装能力：将基础组件、开源组件封装和基础页面模板组合为可运行页面。
- `C3` 场景闭环能力：将页面继续编排为登录等业务流程并保持跨页一致性。
- `C4` 非能力项：不负责后端业务规则、数据库模型与基础设施部署。

## 2. 边界定义（Boundary / 参数）

- `BA` 原子覆盖率：核心场景必须有自研原子组件兜底。来源：`C1`。
- `DS` 设计系统版本：组件样式和交互契约必须绑定统一版本。来源：`C1 + C2`。
- `OS` 开源组件白名单：仅允许引入通过评审的外部组件。来源：`C1 + C2`。
- `PT` 基础页面模板集：至少覆盖登录、注册、找回密码等通用页面。来源：`C2`。
- `RT` 路由层级约束：页面嵌套深度需控制在可维护范围。来源：`C2 + C3`。
- `SLA` 场景完成时延：关键流程从进入到完成需满足时延阈值。来源：`C3`。
- `UX` 交互一致性：同类交互反馈、状态提示和错误表达必须一致。来源：`C1 + C2 + C3`。

## 3. 最小可行基（Minimum Viable Bases）

- `B1` 原子组件基：L1.M0[R1,R2]。来源：`C1 + BA + DS + UX`。
- `B2` 资源装配基：L1.M0[R2]。来源：`C1 + C2 + OS + PT + DS`。
- `B3` 页面编排基：L1.M0[R2,R3]。来源：`C2 + C3 + RT + SLA + UX`。

## 4. 基组合原则（Base Combination Principles）

- `R1` 原子组件标准化
  - `R1.1` 参与基：`B1`。
  - `R1.2` 组合方式：统一交互事件、视觉令牌映射与可访问性属性，沉淀稳定原子目录。
  - `R1.3` 输出能力：`C1`。
  - `R1.4` 边界绑定：`BA/DS/UX`。
- `R2` 基础资源并轨
  - `R2.1` 参与基：`B1 + B2`。
  - `R2.2` 组合方式：对开源组件与基础页面模板进行二次封装，使其遵循原子层接口和主题规范。
  - `R2.3` 输出能力：`C1 + C2`。
  - `R2.4` 边界绑定：`OS/PT/DS/UX`。
- `R3` 页面到流程组装
  - `R3.1` 参与基：`B1 + B2 + B3`。
  - `R3.2` 组合方式：以页面模板为骨架组合原子组件，形成登录等业务页面并接入流程编排。
  - `R3.3` 输出能力：`C2 + C3`。
  - `R3.4` 边界绑定：`BA/OS/PT/RT/SLA/UX`。
- `R4` 禁止组合
  - `R4.1` 参与基：`B1 + B2 + B3`。
  - `R4.2` 组合方式：绕过原子层直接拼接业务页、引入未审核外部组件或违反路由时延约束的方案均无效。
  - `R4.3` 输出能力：`C4`。
  - `R4.4` 边界绑定：`BA/DS/OS/PT/RT/SLA/UX`。

## 5. 验证（Verification）

- `V1` 基础完整性：`B1` 必须覆盖核心输入与点击类交互。
- `V2` 外部一致性：`B2` 中所有外部资源必须满足 `OS` 与 `DS` 约束。
- `V3` 组装覆盖性：`R2~R3` 必须可产出登录类页面并可挂接流程。
- `V4` 约束符合性：所有有效组合必须满足 `RT/SLA/UX` 边界。
- `V5` 最小必要性：移除任一 `B*` 后，`V2~V4` 至少一项失败。
- `V6` 结论表达：逐条输出 `R* -> C* / 边界` 的通过或失败结论。
