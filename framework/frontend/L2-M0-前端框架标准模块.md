# 前端框架标准模块:FrontendFrameworkStandard

@framework

本模块建立在“前端基础原子 -> 前端界面原子 -> 前端标准 -> 页面与运行编排”这条压缩链上。承载绑定、状态交互与视觉语义位于 `frontend/L0-M0 ~ L0-M2`，触发选择、文本输入、展示容器、集合导航与标记反馈位于 `frontend/L1-M0 ~ L1-M4`，本模块只定义前端通用框架如何承接这些结构。项目实例应把本模块编译成显式前端 contract 和规则校验，而不是在实例配置中重写这些结构。对知识问答客户端这类实例，典型形态是“会话侧栏 / 聊天主区 / 按需来源抽屉 + 辅助详情页”。实例配置采用“大类边界 -> 大类 section”约定：`SURFACE -> [surface]`、`VISUAL -> [visual]`、`ROUTE -> [route]`、`A11Y -> [a11y]`；`INTERACT + STATE + EXTEND` 当前仍由框架 contract 固定，不直接开放为实例 section。对应的 `product_spec.toml` 与 `implementation_config.toml` 应默认使用中文详细注释，便于人与 AI 在前端产品形态和实现细化之间进行对齐。本模块中的 `B*` 是当前模块的最小结构承接单元，可以由 `L1` 原子族组合而成，并不要求在全框架引用图里都是全局叶子。本模块的 Markdown 表达、编号与作者入口约束遵循 `specs/框架文档Lint标准.md`，其中 plain `@framework` 直达模板入口属于硬约束。在 `Framework -> Product Spec -> Implementation Config -> Code` 分层中，本模块只负责共同前端结构语言，不直接固化某个产品的最终真相或具体技术栈。

## 1. 能力声明（Capability Statement）

- `C1` 通用界面契约能力：把输入、展示、导航与反馈原子收敛为稳定前端界面骨架。
- `C2` 跨场景交互承接能力：以统一结构承接聊天、来源验证、知识库浏览与详情页跳转等不同场景交互。
- `C3` 领域框架承接能力：为更高层的领域通用框架输出稳定 UI 承接面，而不泄漏底层实现细节。
- `C4` 非能力项：不负责具体领域对象定义、后端存储模型与项目实例值。

## 2. 边界定义（Boundary / 参数）

- `SURFACE` 界面边界：头部、会话侧栏、聊天主区、来源抽屉、辅助页面与反馈区等承载面职责需明确。来源：`C1 + C2`。
- `VISUAL` 视觉边界：颜色、字号级、圆角级、阴影级与密度档位需通过受控 token 与 recipe 表达。来源：`C1 + C3`。
- `INTERACT` 交互边界：输入、选择、切换、展开、创建、删除与提交等交互出口需稳定。来源：`C1 + C2`。
- `STATE` 状态边界：加载、空态、错误与成功等结构状态需统一表达。来源：`C1 + C2`。
- `EXTEND` 扩展边界：领域框架只能在既有插接口上扩展，不得重写通用原子职责。来源：`C2 + C3`。
- `ROUTE` 路由边界：通用页面入口、返回路径与深链落点需可承接。来源：`C2 + C3`。
- `A11Y` 可访问边界：阅读顺序、键盘导航与提示关系需可达。来源：`C1 + C2`。

## 3. 最小可行基（Minimum Viable Bases）

- `B1` 承载交互契约基：L1.M0[R2] + L1.M1[R2] + L1.M2[R1,R2]。来源：`C1 + SURFACE + VISUAL + INTERACT + STATE`。
- `B2` 导航反馈扩展契约基：L1.M3[R1,R2] + L1.M4[R1,R2]。来源：`C2 + C3 + INTERACT + ROUTE + EXTEND + A11Y`。

## 4. 基组合原则（Base Combination Principles）

- `R1` 通用骨架收敛
  - `R1.1` 参与基：`B1`。
  - `R1.2` 组合方式：先固定触发、文本输入与展示容器之间的主关系，再把 surface 与 visual 大类 section 中的实例值编译成稳定 layout recipe 和 token。
  - `R1.3` 输出能力：`C1`。
  - `R1.4` 边界绑定：`SURFACE + VISUAL + INTERACT + STATE`。
- `R2` 场景扩展闭合
  - `R2.1` 参与基：`B1 + B2`。
  - `R2.2` 组合方式：在统一骨架上接入集合导航与标记反馈，约束交互出口、返回路径与可访问顺序，使不同场景共享同一前端结构骨架，并为领域框架提供稳定插接口。
  - `R2.3` 输出能力：`C1 + C2 + C3`。
  - `R2.4` 边界绑定：`INTERACT + ROUTE + A11Y + STATE + EXTEND + SURFACE + VISUAL`。
- `R3` 禁止组合
  - `R3.1` 参与基：`B1 + B2`。
  - `R3.2` 组合方式：让领域框架直接改写通用原子职责、绕过交互契约或破坏承载面边界的方案均无效。
  - `R3.3` 输出能力：`C4`。
  - `R3.4` 边界绑定：`SURFACE + VISUAL + INTERACT + STATE + EXTEND + ROUTE + A11Y`。

## 5. 验证（Verification）

- `V1` 骨架完整性：通用界面必须同时具备触发、输入、展示、导航和反馈承载面。
- `V2` 交互一致性：不同场景必须共享统一交互出口与返回结构。
- `V3` 扩展稳定性：领域扩展必须通过 `EXTEND` 插接口完成，不得破坏通用职责。
- `V4` 可访问符合性：有效组合必须满足 `A11Y` 约束。
- `V5` 最小必要性：移除任一 `B*` 后，`V2~V4` 至少一项失败。
- `V6` 配置映射稳定性：`[surface]/[visual]/[route]/[a11y]` 中的字段必须只落到对应大类边界，不得跨大类游离。
- `V7` 结论表达：逐条输出 `R* -> C* + 边界绑定` 的通过或失败结论。
