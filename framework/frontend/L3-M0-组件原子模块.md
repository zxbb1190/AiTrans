# 组件原子模块:ComponentAtoms

@framework

## 1. 能力声明（Capability Statement）

- `C1` 原子抽象能力：将基础 UI 组件沉淀为稳定可复用单元。
- `C2` 交互契约能力：统一事件、状态与可访问性行为。
- `C3` 扩展发布能力：支持主题扩展、文档化和版本迭代。
- `C4` 非能力项：不负责路由策略、页面业务流程与后端接口编排。

## 2. 边界定义（Boundary / 参数）

- `ATOM` 原子组件清单：至少覆盖 Button、Input、Select、Checkbox。来源：`C1`。
- `API` 组件接口稳定等级：破坏性改动必须走版本升级。来源：`C1 + C2`。
- `A11Y` 可访问性约束：键盘可达、语义标签和提示文案必须完整。来源：`C2`。
- `THEME` 主题令牌映射：颜色、间距、字号和圆角需可配置。来源：`C1 + C3`。
- `TEST` 自动化测试阈值：关键交互路径必须具备回归测试。来源：`C2 + C3`。
- `I18N` 国际化约束：默认文案和占位提示需可替换。来源：`C3`。
- `DOC` 文档完备度：每个原子组件需有参数与示例说明。来源：`C1 + C3`。

## 3. 最小可行基（Minimum Viable Bases）

- `B1` 交互原子基：L2.M0[R1,R2]。来源：`C1 + C2 + ATOM + API + A11Y`。
- `B2` 展示原子基：L2.M0[R2]。来源：`C1 + C2 + ATOM + THEME + I18N`。
- `B3` 发布封装基：L2.M0[R2,R3]。来源：`C2 + C3 + API + TEST + DOC`。

## 4. 基组合原则（Base Combination Principles）

- `R1` 原子目录构建
  - `R1.1` 参与基：`B1 + B2`。
  - `R1.2` 组合方式：先确定组件清单，再统一交互语义、样式映射和状态反馈表达。
  - `R1.3` 输出能力：`C1 + C2`。
  - `R1.4` 边界绑定：`ATOM/API/A11Y/THEME`。
- `R2` 规范封装发布
  - `R2.1` 参与基：`B1 + B2 + B3`。
  - `R2.2` 组合方式：将原子组件接入文档与测试流水线，并按版本策略对外发布。
  - `R2.3` 输出能力：`C2 + C3`。
  - `R2.4` 边界绑定：`API/TEST/DOC/I18N/THEME`。
- `R3` 页面可组装契约输出
  - `R3.1` 参与基：`B1 + B2 + B3`。
  - `R3.2` 组合方式：输出统一 props 约定和事件契约，供页面层直接编排登录等页面。
  - `R3.3` 输出能力：`C1 + C2 + C3`。
  - `R3.4` 边界绑定：`ATOM/API/A11Y/TEST/DOC`。
- `R4` 禁止组合
  - `R4.1` 参与基：`B1 + B2 + B3`。
  - `R4.2` 组合方式：缺失基础交互组件、无测试发布链路或与主题令牌脱节的组件集合均无效。
  - `R4.3` 输出能力：`C4`。
  - `R4.4` 边界绑定：`ATOM/API/A11Y/THEME/TEST/I18N/DOC`。

## 5. 验证（Verification）

- `V1` 清单覆盖：`ATOM` 必须覆盖核心输入与反馈组件。
- `V2` 契约一致：`R1~R2` 生成的接口和事件语义必须一致。
- `V3` 质量约束：`TEST` 与 `DOC` 必须达到设定阈值。
- `V4` 边界符合：所有有效组合需满足 `A11Y/THEME/I18N` 约束。
- `V5` 最小必要性：移除任一 `B*` 后，`V2~V4` 至少一项失败。
- `V6` 结论表达：逐条输出 `R* -> C* / 边界` 的通过或失败结论。
