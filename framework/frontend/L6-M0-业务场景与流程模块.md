# 业务流程场景模块:BusinessFlowScenarios

@framework

## 1. 能力声明（Capability Statement）

- `C1` 场景流程编排能力：将页面串联为完整业务场景流程。
- `C2` 跨页状态一致能力：保持会话、鉴权和中间结果在流程内一致。
- `C3` 过程观测与回滚能力：支持关键节点监控、告警与快速回退。
- `C4` 非能力项：不负责组件级样式定义与底层路由实现细节。

## 2. 边界定义（Boundary / 参数）

- `SCENE` 场景清单：至少覆盖登录、注册、找回密码和登出。来源：`C1`。
- `STEP` 流程步骤约束：每个场景的关键步骤需可枚举与可追踪。来源：`C1 + C2`。
- `AUTH` 鉴权策略：鉴权前置条件、令牌校验和失效处理需统一。来源：`C2`。
- `SESSION` 会话策略：会话生命周期、续期和失效清理需明确。来源：`C2`。
- `FALLBACK` 异常回退策略：失败场景需定义降级与重试入口。来源：`C2 + C3`。
- `METRIC` 关键指标阈值：转化率、失败率和耗时指标需可观测。来源：`C1 + C3`。
- `ROLLBACK` 发布回滚机制：流程变更需支持开关关闭和快速回滚。来源：`C3`。

## 3. 最小可行基（Minimum Viable Bases）

- `B1` 场景骨架基：L5.M0[R1,R2,R3]。来源：`C1 + SCENE + STEP + AUTH`。
- `B2` 跨页状态基：L5.M0[R2]。来源：`C2 + AUTH + SESSION + FALLBACK`。
- `B3` 流程观测基：L5.M0[R2,R3]。来源：`C2 + C3 + METRIC + ROLLBACK + FALLBACK`。

## 4. 基组合原则（Base Combination Principles）

- `R1` 主链路场景编排
  - `R1.1` 参与基：`B1 + B2`。
  - `R1.2` 组合方式：将入口页、验证页和结果页按顺序连接，并对每一步绑定会话与鉴权状态。
  - `R1.3` 输出能力：`C1 + C2`。
  - `R1.4` 边界绑定：`SCENE/STEP/AUTH/SESSION`。
- `R2` 异常链路编排
  - `R2.1` 参与基：`B2 + B3`。
  - `R2.2` 组合方式：对超时、鉴权失败和提交失败等异常路径配置回退、告警与重试机制。
  - `R2.3` 输出能力：`C2 + C3`。
  - `R2.4` 边界绑定：`FALLBACK/METRIC/ROLLBACK/SESSION`。
- `R3` 登录场景闭环编排
  - `R3.1` 参与基：`B1 + B2 + B3`。
  - `R3.2` 组合方式：将登录场景主链路与异常链路合并，形成可监控、可回退的完整流程。
  - `R3.3` 输出能力：`C1 + C2 + C3`。
  - `R3.4` 边界绑定：`SCENE/STEP/AUTH/SESSION/FALLBACK/METRIC`。
- `R4` 禁止组合
  - `R4.1` 参与基：`B1 + B2 + B3`。
  - `R4.2` 组合方式：缺失会话一致性、未定义失败回退或缺少指标监控的流程均无效。
  - `R4.3` 输出能力：`C4`。
  - `R4.4` 边界绑定：`SCENE/STEP/AUTH/SESSION/FALLBACK/METRIC/ROLLBACK`。

## 5. 验证（Verification）

- `V1` 场景覆盖性：`SCENE` 中每个场景必须具备主链路定义。
- `V2` 状态一致性：主链路与异常链路中的会话状态必须可追踪且一致。
- `V3` 回退有效性：`FALLBACK` 中每类异常必须有可执行回退路径。
- `V4` 观测符合性：关键节点必须输出 `METRIC` 并满足阈值约束。
- `V5` 最小必要性：移除任一 `B*` 后，`V2~V4` 至少一项失败。
- `V6` 结论表达：逐条输出 `R* -> C* / 边界` 的通过或失败结论。
