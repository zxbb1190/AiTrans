# 最终架构重构验收标准

## 0. 文档定位

本文件把“全仓彻底重构到最终新架构”的提示词固化成仓库内正式审查基准。  
后续如果需要判断仓库是否已经达到最终纯态，应直接以本文件为对照标准，而不是再回到聊天提示词逐段比对。

本文件的用途只有两个：

1. 作为重构目标的正式文字基准
2. 作为后续代码审查、架构审查、生成链审查的直接依据

## 1. 禁止项

出现以下任一情况，都不能判定为“已经完成最终重构”：

- 保留旧架构作为并行主路径，再在旁边追加新架构
- 长期保留兼容层、适配层、过渡层作为正式入口
- 只做到 MVP、临时版、半套重构
- 留下关键 TODO，声称之后再收口
- 保留多个 JSON / manifest / tree / report 继续分别承担不同部分的机器真相
- 保留旧的项目级大编排器，只是换名词或换文件名
- 用文档叙事宣称已完成，但核心代码路径仍是旧方案

## 2. 最终目标

仓库必须完整收敛到如下主链：

```text
Framework markdown
  -> framework parser / loader
    -> framework tree / module tree
      -> registry lookup
        -> package entry classes
          -> config slicing / package compile
            -> package exports
              -> assembled code runtime
                -> evidence generation
                  -> canonical JSON
                    -> derived governance / tree / report / views
```

只有当仓库的编译链、运行链、验证链、生成链和文档叙事都统一到这条主链时，才算完成。

## 3. 架构原则

### 3.1 Markdown-first

- `framework/*.md` 必须继续是 framework 层作者源
- 不允许把 framework 真相源迁移成 schema/config，再反向生成 markdown
- 必须是先有 markdown，再有 parser/loader，再有 package 注册与装配

### 3.2 One framework file -> one package

- 每个 framework 文件必须对应一个独立代码 package
- package 可以有多个内部文件，但只能有一个正式入口 class
- package 必须自成体系、可独立理解、可独立测试、可拆卸

### 3.3 One package -> one entry class

- 每个 package 必须有且只有一个正式入口 class
- 只有入口 class 可以注册到主 registry
- helper / schema / adapter / internal builder 都不能直接进入主注册链

### 3.4 Unified contract

每个入口 class 必须实现统一 Framework Package Contract。语义上至少要明确：

- 对应哪份 framework 文件
- 模块 id 是什么
- 需要哪些配置切片
- 可以组合哪些 child packages
- 如何 compile
- 如何 export
- 如何贡献 evidence

### 3.5 Explicit registry

必须存在统一 registry，并把它作为正式绑定真相。registry 至少要承担：

- framework 文件 -> entry class 绑定
- module id -> entry class 绑定
- package 冲突检测
- framework 未实现检测
- orphan package 检测
- 编译器按模块树查询 package
- validator 检查 package / framework 一一对应关系
- canonical JSON 记录正式 registry binding

### 3.6 Composition only

- package 装配关系只能通过组合表达
- 上层只能通过 `import 子 package + 分发 config slice + 调用 compile / export` 组合下层
- 严禁用继承树表达 framework/module 关系

### 3.7 Config-driven projects

项目必须由以下三者共同决定：

- framework 树 / 模块树选择
- 项目配置内容
- registry 中可用 package

项目差异应主要通过模块树与配置表达，而不是通过项目级手写特化大编排器表达。

### 3.8 Config merge but logical partition

项目配置必须统一到一套体系中，但逻辑上至少清晰区分：

- `selection`
- `truth`
- `refinement`
- `narrative`

自然语言说明只能用于解释，不得替代机器判定字段。

### 3.9 Strict config slicing

- 配置必须按模块树逐层切片分发
- 每个 package 必须声明自己的 config contract
- package 不得偷偷消费未声明字段
- 不允许全局配置字典任意透传

### 3.10 Single canonical JSON

- 最终机器真相源只能有一个 canonical JSON
- GUI、validator、change propagation、coverage、report generation 必须以 canonical JSON 为唯一机器基础
- 其他 manifest / tree / report 如果保留，只能是 canonical JSON 的派生视图

## 4. 目标架构分层

### 4.1 Framework layer

- 作者源：`framework/*.md`
- parser / loader 负责把 markdown 解析成可装配模块语义
- framework 真相源不得转移出 markdown

### 4.2 Package layer

- 每个 framework 模块对应一个 package
- package 内唯一入口 class 实现 Framework Package Contract
- package 对外只有入口 class 的 contract 具有正式地位

### 4.3 Config layer

- 项目配置统一在 `projects/<project_id>/project.toml`
- 必须能表达模块树选择、产品真相、实现细化、说明文字
- 配置必须可被逐层切片分发给 package

### 4.4 Compile / assembly layer

编译器核心职责必须是：

- 读取 framework 树
- 读取项目配置
- 从 registry 找到 package
- 校验 config contract
- 装配 child packages
- 调用 compile / export
- 汇总 canonical JSON

不得继续保留旧的项目级大聚合模块作为编译核心。

### 4.5 Evidence layer

Evidence 必须作为正式主层存在，负责承载：

- 编译证据
- 运行证据
- 结构证据
- 审计 / 治理派生视图来源

但 Evidence 不得成为新的平行真相源，正式机器落点仍然必须回到 canonical JSON。

## 5. Framework Package Contract 验收要求

每个 package 的正式入口 class，必须能稳定回答：

1. 它对应哪份 framework 文件
2. 它的模块 id 是什么
3. 它允许消费哪些配置字段
4. 它允许组合哪些 child packages
5. 给定 config slice 和 child compile result 后如何 compile
6. compile 后向上层暴露什么 export
7. 如何纳入统一 evidence 汇总链

## 6. Config 体系验收要求

旧的 Product / Implementation 双轨体系必须彻底重构为统一项目配置体系。验收要求：

- 不再保留两套平行大对象模型作为主路径
- 物理可合并，逻辑必须分区明确
- 支持模块树逐层切片
- 每个 package 的 config contract 能用于自动校验
- 不允许 package 跳过自己的切片直接偷读全局配置
- 不允许说明文字成为机器唯一判定依据

如果仓库存在多个项目发现机制，必须统一切换到新配置体系，而不是只改示例项目。

## 7. Canonical JSON 验收要求

canonical JSON 必须稳定记录至少以下内容：

- framework tree / module tree
- registry binding
- config projection
- compile result
- code / runtime projection
- evidence
- derived view metadata

并且其分层必须清晰区分：

- `framework`
- `config`
- `code`
- `evidence`

## 8. 旧实现清理要求

以下旧机制如存在，必须删除、替换或明确降级为 canonical 的派生视图，不得继续作为主路径：

- 旧项目级大聚合模块体系
- 旧平行真相文件
- 旧治理主路径
- 旧绑定方式
- 旧架构文档叙事

## 9. 审查清单

后续审查应至少逐项回答以下问题：

1. `framework/*.md` 是否仍是作者源？
2. 每个 framework 文件是否已经对应一个 package？
3. 每个 package 是否只有一个唯一入口 class？
4. 每个入口 class 是否实现统一 contract？
5. 每个入口 class 是否已经注册到统一 registry？
6. registry 是否已经成为 framework 与代码的一一绑定真相？
7. Product / Implementation 是否已经重构为统一项目配置体系？
8. config slicing 是否按模块树逐层分发？
9. package 是否只消费自己声明的配置？
10. 编译器核心是否已从项目级大聚合模块转为模块树 + registry + config + package compile？
11. canonical JSON 是否已成为唯一机器真相源？
12. 其他 manifest / tree / report 是否都已经降级为 derived views？
13. Evidence 是否已经纳入正式主层但未形成新的平行真相源？
14. 旧架构核心是否已经被删除、替换或彻底降级？
15. 文档、脚本、生成物、运行时装配是否都已经统一到新架构语言？

只要以上任一条不能明确回答“是”，就不能判定为“最终重构完成”。

## 10. 使用方式

后续若要发起审查，可直接引用本文件并使用类似要求：

```text
请对照 docs/最终架构重构验收标准.md 审查当前仓库，
判断哪些条目已满足、哪些条目仍未满足，并给出对应代码位置。
```
