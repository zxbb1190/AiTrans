# Shelf 方案总览与简化评估

## 1. 文档目的

这份文档不再按单个文件或单个功能去描述 `shelf`，而是把当前仓库当成一个整体系统来整理。

目标有两个：

1. 说明 `shelf` 现在到底在做什么，真实主线是什么。
2. 从更高维度判断：这套方案是否已经是当前最好的、最简洁的实现方式；如果不是，哪里还能继续压缩，而且不损失核心效果。

## 1.1 当前状态说明

这份文档最初作为重构评估写成；其中提出的第一、第二、第三阶段收敛已经开始落地。

当前已完成的关键动作包括：

- 默认架构文档已切换到“项目编译链”主线；
- legacy shelf 样本入口已下沉到 `src/examples/legacy_shelf/`，对应文档和证据迁移到 `docs/legacy_shelf/`；
- `template_registry` 已从主运行链移除，当前主链明确按单模板知识库编译器运行；
- `mapping_registry.json` 的 `tree` 已改为 canonical 标准树同步视图，不再被视为独立人工真相源。

---

## 2. 一句话定义当前 Shelf

当前的 `shelf`，本质上已经不是一个单纯的“置物架求解项目”。

它更准确的定义是：

> 一个以“结构”为第一性对象的框架文档编译仓库。

它试图把下面这条链打通：

> `Framework -> Product Spec -> Implementation Config -> Code -> Evidence`

也就是说，这个仓库真正想证明的不是“某个页面怎么写”或“某个置物架怎么算”，而是：

- 框架文档能否成为人和 AI 共享的结构语言；
- 这种结构语言能否约束产品真相；
- 产品真相能否再细化为实现路径；
- 这条路径能否物化成运行时代码、治理产物与验证证据；
- 以上过程能否保持单向、可追溯、可验证。

---

## 3. 当前方案的真实结构

从仓库当前内容看，`shelf` 实际上由三条叙事线组成。

### 3.1 主线一：框架标准语言

这一层是仓库的思想核心。

- `specs/规范总纲与树形结构.md`
- `specs/框架设计核心标准.md`
- `specs/框架文档Lint标准.md`
- `framework/*/*.md`

这一层回答的是：

- 什么是结构；
- 什么是能力、边界、最小可行基、组合原则、验证；
- 领域框架如何从更基础结构逐层展开；
- 框架文档怎样表达才能被机器读取和验证。

这是仓库最重要的部分，也是整个系统存在的理由。

### 3.2 主线二：框架文档到项目运行时的编译链

这一层是仓库的工程主线。

核心链路大致如下：

1. `framework/*.md` 被解析为机器可读 IR。
   - 入口：`src/framework_ir/parser.py`
2. 框架树、模块关系、表达协议、引用闭合性、弱充分性被校验。
   - 入口：`scripts/validate_strict_mapping.py`
3. 具体项目通过 `product_spec.toml` 和 `implementation_config.toml` 固定产品真相与实现路径。
   - 示例：`projects/knowledge_base_basic/`
4. 编译器把框架、产品规格和实现配置收敛成项目运行时描述与生成产物。
   - 入口：`src/project_runtime/knowledge_base.py`
   - 调度：`scripts/materialize_project.py`
5. 最终生成运行时 bundle、治理清单、治理树、产品 JSON 等证据层产物。
   - 落点：`projects/<project_id>/generated/*`

这一层说明：`shelf` 已经不是只写标准，它在做“结构化文档编译”。

### 3.3 主线三：历史置物架参考样本

这一层来自仓库最早的原型方向，仍然保留着完整的“置物架求解/验证/可视化”链路。

- `src/examples/legacy_shelf/reference_pipeline.py`
- `src/domain/`
- `src/enumeration/`
- `src/geometry/`
- `src/metrics/`
- `src/verification/`
- `src/visualization/`

这条线的价值在于：

- 它证明“框架语言”不是纯口号，确实可以落到一个严格对象域；
- 它保留了从抽象规则到枚举、验证、证据输出的完整示范；
- 它是仓库方法论的重要来源。

但从当前默认入口看，这条线已经不是仓库的主运行路径。它现在作为 legacy 样本保留，默认运行入口仍然是项目驱动的知识库 demo。

---

## 4. 当前方案的真实主链

如果只看今天仓库默认会走哪条路，那么主链其实已经很清楚：

1. 框架文档定义共同结构语言。
2. 项目产品规格固定“这个产品最终是什么”。
3. 实现配置固定“这次用哪条技术路径落地”。
4. 编译器把三者物化为运行时 bundle 和治理证据。
5. FastAPI runtime 把这个 bundle 暴露为可运行产品。

也就是说，当前 `shelf` 的主产品已经不是“置物架”，而是“一个由框架文档编译出来的知识库工作台示例”。

这件事必须说清楚，否则读者会同时收到两种信号：

- 仓库名字叫 `shelf`，像是在做置物架；
- 默认运行入口却是在跑 knowledge base。

这不是小问题，而是理解成本的主要来源之一。

---

## 5. 从更高维度看：这套方案的真实本体是什么

如果把所有具体领域都拿掉，只保留结构关系，那么当前方案的本体其实可以压成一句话：

> 这是一个“结构语言编译器 + 项目物化器 + 治理验证器”的组合系统。

从这个更高维度看，仓库中的很多东西就能被重新归类：

- `framework/*.md` 不是普通文档，而是源语言。
- `framework_ir` 不是辅助工具，而是编译中间表示。
- `product_spec.toml` 不是普通配置，而是产品真相层输入。
- `implementation_config.toml` 不是普通配置，而是实现路径层输入。
- `generated/*` 不是输出缓存，而是证据层产物。
- `validate_strict_mapping.py` 不是格式检查脚本，而是结构合法性验证器。

一旦把它看成“编译系统”，很多设计就变得合理了：

- 为什么要坚持层级与映射；
- 为什么要强调单向真相流；
- 为什么要把 `Framework` 和 `Product Spec` 分开；
- 为什么要物化 `generated/*`；
- 为什么要做治理清单和治理树。

所以，方向本身没有问题。问题不在“要不要有这套系统”，而在“这套系统里哪些复杂度是真必要，哪些只是当前实现阶段叠出来的”。

---

## 6. 当前方案中真正必要的复杂度

下面这些复杂度，我认为是必要的，不能为了“看起来更简单”就拿掉。

### 6.1 Framework / Product Spec / Implementation Config 的分层

这是仓库最关键的结构分层。

一旦把这三层重新混回一个文件或一种语义里，仓库最重要的价值就会丢失：人和 AI 无法再清晰讨论“共同结构”“产品真相”“实现细化”分别是什么。

### 6.2 框架文档可解析、可 lint、可验证

如果 `framework/*.md` 只是给人看而不能进机器，整个“共同结构语言”就会退回宣言文本。

所以：

- 统一的 Markdown 协议；
- 可解析的 section；
- 可校验的 `C/B/R/V`；
- 引用图合法性；
- 弱充分性检查；

这些都属于必要复杂度。

### 6.3 generated 证据层

只保留源而不产出证据，会让“框架到实现”的链路停在口头承诺上。

因此，生成：

- `framework_ir.json`
- `product_spec.json`
- `implementation_bundle.py`
- `generation_manifest.json`
- 治理清单和治理树

这件事本身是合理的。

### 6.4 治理与可追溯能力

这个仓库不是纯 demo，而是把“约束”“证据”“传导”“验证”当成核心能力。

因此，存在治理层不是问题；真正需要警惕的是治理层有没有重复表达同一件事。

---

## 7. 重构前偏高的复杂度

下面这些地方，我认为不是“理念必须”，而是当前实现仍可继续压缩的部分。

### 7.1 仓库同时保留两套主叙事，认知成本偏高

今天的仓库同时存在：

- “结构化框架编译器”叙事；
- “置物架求解与参考验证”叙事。

这两条线都不是错，但并排放在默认入口附近，会让读者很难判断什么是主线、什么是样本、什么是历史资产。

这类复杂度不是结构复杂度，而是叙事复杂度。

### 7.2 `mapping/mapping_registry.json` 仍然承担了过重的人工维护责任

当前 registry 很重要，但它同时承载了两种信息：

1. 标准树
2. 标准到实现的映射

问题在于：

- 标准树本身已经部分存在于文件系统和框架文档结构中；
- 实现映射又部分存在于代码注解、治理扫描与生成脚本中。

这会造成多源维护压力。

更高维度看，registry 应更像“编译产物”或“半生成清单”，而不应长期承担这么重的手写真相职责。

### 7.3 模型层有重复表达

当前至少存在几套相邻但不完全相同的模型：

- `src/framework_core.py`
- `src/framework_ir/models.py`
- `src/shelf_domain.py`
- `src/project_runtime/knowledge_base.py` 里的项目级 dataclass

这不是完全错误，因为它们服务的层次不同。

但从当前使用情况看，`framework_core.py` 和主链的关系已经较弱，更像历史兼容层或参考样本支持层。

如果长期保留，会继续抬高理解成本。

### 7.4 `template_registry` 曾经有“预支抽象”的味道

`src/project_runtime/template_registry.py` 的抽象本身不差，它是在为多模板项目物化做准备。

但从当前仓库状态看，真正内建并跑通的模板主要还是 `knowledge_base_workbench` 这一条。

因此，这个抽象目前还没有完全赚回自己的复杂度成本。

如果近期不会出现多个成熟模板，那么它就是“方向合理但时机偏早”的抽象。

### 7.5 治理产物有一定重复

现在同时存在：

- project 级治理 manifest / tree
- workspace 级治理 tree / html

这些能力都合理，但从数据建模角度看，仍有机会收敛为“一份规范化图数据 + 多种视图输出”，而不是多份相近结构并行存在。

### 7.6 `docs/architecture.md` 曾经不再代表默认主线

当前 `docs/architecture.md` 主要描述的是 legacy shelf reference pipeline。

但仓库默认主线已经转到：

> framework 文档 -> product spec -> implementation config -> materialize -> runtime app

这意味着仓库已有文档叙述和默认行为之间出现偏移。

这不是代码问题，而是信息架构问题。

---

## 8. 结论：当前方案是不是目前最好的、最简洁的实现方法

我的判断是：

> 方向是对的，但还不是当前最简实现。

更准确地说，需要分两层回答。

### 8.1 如果目标是“证明这套方法论能贯通到运行时代码”

那么当前方案是合理的，甚至可以说已经接近一个完整原型。

它已经具备：

- 结构语言；
- 项目真相分层；
- 编译中间表示；
- 自动物化；
- 治理验证；
- 可运行样例；
- 证据层产物。

从“方法论闭环”角度，它是成立的。

### 8.2 如果目标是“用最少结构实现同样效果”

那么当前方案还可以明显压缩。

最主要的冗余不在思想上，而在下面三类地方：

- 多条叙事线并存；
- 多份手写注册与治理表达并存；
- 多层概念模型并存。

所以答案不是“推倒重来”，而是：

> 保留当前思想骨架，压缩重复表达层。

---

## 9. 一个更简洁、但能保持同等效果的实现方案

如果目标是不牺牲现有效果，只追求更简洁，那么我认为可以收敛到下面这个版本。

### 9.1 只保留三类人工源

把人工维护的第一性输入严格压到三类：

1. `specs/*` + `framework/*.md`
2. `projects/*/product_spec.toml`
3. `projects/*/implementation_config.toml`

除此之外，其它都尽量生成。

### 9.2 把 registry 从“手写真相”降级为“生成清单”或“半生成清单”

建议方向有两个，至少选一个：

- 方案 A：`mapping/mapping_registry.json` 完全由框架解析和代码注解扫描生成。
- 方案 B：保留少量手写映射字段，但把标准树和可推导部分全部改为生成。

这样可以显著降低“文档改了，registry 还要手动跟”的同步成本。

### 9.3 把 legacy shelf reference pipeline 下沉为样本区

建议把这条线明确降级为：

- `examples/legacy_shelf/`
- 或 `src/examples/legacy_shelf/`

它仍然保留，但不再和当前主运行链并列争夺“仓库主叙事”。

这不会损失价值，反而会让仓库主线更清楚。

### 9.4 收敛模型层

建议原则是：

- 主链只保留“框架 IR 模型 + 项目编译模型 + runtime 模型”。
- 参考样本需要的旧模型，明确归到样本区。

这样 `framework_core.py` 这种中间层就不会继续在主链里制造歧义。

### 9.5 只有在出现第二个成熟模板后，才保留 `template_registry` 的完整抽象

如果近期仍只有一个主模板，那么更简洁的做法是：

- 直接把 `knowledge_base` 作为明确编译目标；
- 等第二个成熟模板出现后，再把注册层抽象提升为仓库默认设施。

如果你坚持保留 `template_registry`，也可以，但应明确把它视为“未来扩展点”，而不是当前系统的核心价值。

### 9.6 治理数据只保留一份 canonical graph

建议把治理层收敛为：

- 一份 canonical JSON 图；
- 其它 manifest、tree、html 都由它派生。

这样同样能保留追溯与可视化能力，但重复表达会少很多。

---

## 10. 我认为最稳的收敛方案

如果不想大改，我建议按下面顺序收敛。

### 第一阶段：先澄清主线，不动核心功能

- 新增总览文档，明确当前主线和 legacy 样本的关系。
- 把默认入口文档改成“项目编译链优先”的叙述。
- 明确 `reference-shelf` 是样本，不是默认主产品。

这一阶段收益最高，风险最低。

### 第二阶段：减少重复真相源

- 把 `mapping_registry.json` 里可推导部分改为生成。
- 收敛治理产物为单一 canonical graph。

这一阶段能明显降低维护成本。

### 第三阶段：压缩工程抽象

- 若近期仍只有一个模板，弱化或折叠 `template_registry`。
- 把 legacy shelf reference pipeline 下沉到样本区。
- 收敛重复模型层。

这一阶段完成后，仓库会更像一个真正清晰的“结构语言编译器”。

---

## 11. 最终判断

最终我的判断可以压成三句话：

1. `shelf` 当前方案的方向是对的，核心思想没有问题。
2. 它已经能证明“结构化框架语言 -> 产品真相 -> 实现路径 -> 运行时与证据”这条链是可操作的。
3. 但它还不是最简实现；最值得压缩的不是框架思想本身，而是重复的注册层、治理表达层、模型层，以及 legacy 样本和主线并存造成的认知分叉。

如果只保留一句最重要的话，那就是：

> 当前 `shelf` 最应该做的，不是推翻方案，而是把“哪些是第一性真相、哪些只是派生产物”再收得更硬一些。
