# 知识库工作台链路对齐说明

## 1. 结论

当前 `projects/knowledge_base_basic/` 这条链路是对得上的。

如果需要看“从某个具体 framework / product / implementation 配置一路如何落到具体代码，以及代码改动后如何反向校验”的细节说明，见：[框架到代码映射与反查覆盖说明.md](/home/xue/code/shelf/docs/框架到代码映射与反查覆盖说明.md)。

它不是“框架文档写一套、代码手写另一套”，而是已经形成了清晰的单向收敛链：

`Framework -> Product Spec -> Implementation Config -> Generated Artifacts -> Runtime Code -> Governance Evidence`

判断“对得上”的依据不是主观描述，而是以下几类可核对证据同时成立：

- 框架层明确声明了前端、知识库场景、后端接口三类共同结构语言。
- `product_spec.toml` 只实例化产品真相，不反向定义实现手段。
- `implementation_config.toml` 只细化实现路径与产物契约，不改写产品真相。
- 生成入口只有一条：`project_runtime.knowledge_base.materialize_knowledge_base_project`。
- 运行时代码中的高风险行为点都被治理清单绑定到了上游 `framework / product_spec / implementation_config`。
- 生成产物和治理树都能反向证明每个关键 symbol 从哪里来。

## 2. 收敛链总览

### 2.1 Framework

知识库项目当前直接依赖三份 `L2` 标准模块：

- `framework/frontend/L2-M0-前端框架标准模块.md`
- `framework/knowledge_base/L2-M0-知识库工作台场景模块.md`
- `framework/backend/L2-M0-知识库接口框架标准模块.md`

它们分别约束：

- 前端共同承载面、视觉/路由/可访问边界与扩展接口
- 知识库 chat-first 工作台的会话、引用、上下文与返回闭环
- 文件/预览/对话三类接口契约与治理边界

`projects/knowledge_base_basic/generated/generation_manifest.json` 已把这三份框架输入和它们向下解析出来的模块闭包记录为正式证据，而不是运行时代码里隐式写死。

### 2.2 Product Spec

`projects/knowledge_base_basic/product_spec.toml` 负责把“结构空间”实例化成具体产品真相。

这份产品规格没有混入实现层决定，而是按边界落到对应 section：

- 前端外层形态：`[surface] [visual] [route] [a11y] [showcase_page]`
- 领域产品真相：`[library] [preview] [chat] [context] [return]`
- 产品内容种子：`[[documents]]`

这一步的意义是：

- 框架只规定“什么结构可能成立”
- 产品规格规定“这个项目最终选中了哪一个产品”

### 2.3 Implementation Config

`projects/knowledge_base_basic/implementation_config.toml` 只做实现路径细化：

- `[frontend]` 选前端 renderer / style / script profile
- `[backend]` 选接口 renderer / transport / retrieval strategy
- `[evidence]` 选证据暴露方式
- `[artifacts]` 明确本项目 canonical 生成物名字

这一层没有重新命名页面、也没有改写引用闭环语义，因此仍然遵守“产品真相在上，实现细化在下”的分层。

## 3. 从配置到生成产物

`src/project_runtime/knowledge_base.py` 是当前知识库模板的编译入口。它做了几件关键事情：

1. 校验 `product_spec.toml` 的模板 id 必须是 `knowledge_base_workbench`
2. 加载并校验 `product_spec.toml` / `implementation_config.toml` 的 layout
3. 解析三份框架标准模块及其下游模块闭包
4. 把产品真相与实现细化编译成统一 `KnowledgeBaseProject`
5. 物化 6 个 canonical 生成物

当前 canonical 生成物只有这 6 个：

- `framework_ir.json`
- `product_spec.json`
- `implementation_bundle.py`
- `generation_manifest.json`
- `governance_manifest.json`
- `governance_tree.json`

旧时代残留的 `project_bundle.py` 和 `workbench_spec.json` 不再属于知识库链路。现在物化阶段会主动清理这类过时产物，lint 也会把多余生成物视为失败。

## 4. 从生成产物到运行时代码

这条链不是“生成完就没人用”。生成物会继续驱动运行时代码。

### 4.1 入口与应用装配

- `src/main.py`
- `src/project_runtime/app_factory.py`
- `src/knowledge_base_runtime/app.py`

其中：

- `main.py` 负责选择产品规格文件并启动服务
- `app_factory.py` 负责把 `product_spec.toml` 交给知识库编译入口
- `knowledge_base_runtime/app.py` 负责把编译结果装配成 FastAPI 路由和页面入口

### 4.2 结构合同生成

有两份框架侧合同先被显式编出来：

- `src/frontend_kernel/contracts.py`
  - `kb.frontend.surface_contract`
- `src/knowledge_base_framework/workbench.py`
  - `kb.workbench.surface_contract`

它们不是直接手抄 Markdown，而是从 `KnowledgeBaseProject` 抽取：

- 哪些承载面存在
- 哪些交互动作存在
- 哪些状态/返回/引用规则存在
- 这些合同属于哪一个 framework module、哪些 `B* / R*`

### 4.3 UI 与 Backend 规格生成

`src/project_runtime/knowledge_base.py` 继续把编译结果收敛成两类实现侧规格：

- `kb.ui.surface_spec`
- `kb.backend.surface_spec`

这两类规格分别被前端与后端 runtime 消费，而不是让 runtime 重新推导一次产品真相。

### 4.4 运行时消费

最终运行时代码消费这些编译结果：

- `src/knowledge_base_runtime/frontend.py`
  - 使用 UI spec / workbench contract 生成聊天页、知识库列表页、详情页、引用抽屉
- `src/knowledge_base_runtime/backend.py`
  - 使用 backend spec / documents / return policy 暴露 API 与回答行为
- `src/knowledge_base_runtime/app.py`
  - 使用 route contract 把页面与 API 实际挂到路径上

换句话说，代码不是主真相源；代码是在消费编译出的项目结构。

## 5. 治理证据如何证明“对得上”

真正关键的是 `generated/governance_manifest.json` 和 `generated/governance_tree.json`。

它们已经把 8 个高风险 symbol 的上游来源全部列出来：

| Symbol | 绑定代码 | 直接上游 |
| --- | --- | --- |
| `kb.runtime.page_routes` | `src/knowledge_base_runtime/app.py` | frontend `R1/R2/R3` + product `route` + implementation `evidence` |
| `kb.frontend.surface_contract` | `src/frontend_kernel/contracts.py` | frontend `R1/R2/R3` + product `surface/route/a11y/library/preview/chat/return` |
| `kb.workbench.surface_contract` | `src/knowledge_base_framework/workbench.py` | knowledge_base `R1/R2/R3/R4` + product `library/preview/chat/context/return/surface` |
| `kb.ui.surface_spec` | `src/project_runtime/knowledge_base.py` | frontend `R1/R2/R3` + product `surface/route/showcase_page/chat/return` |
| `kb.backend.surface_spec` | `src/project_runtime/knowledge_base.py` | backend `R1/R2/R3/R4` + product `library/chat/context/return/route` + implementation `backend` |
| `kb.api.library_contracts` | `src/knowledge_base_runtime/backend.py` | backend `R1/R2/R3/R4` + product `route/library/preview` |
| `kb.api.chat_contract` | `src/knowledge_base_runtime/backend.py` | backend `R1/R2/R3/R4` + product `route/chat/context/return` |
| `kb.answer.behavior` | `src/knowledge_base_runtime/backend.py` | knowledge_base `R1/R2/R3/R4` + product `chat/context/return` + implementation `backend` |

这张表的意义是：

- 不是只有“这个 symbol 属于哪个文件”
- 而是每个关键 symbol 都有上游 `rule / section` 闭包
- 因此可以从代码反追到框架和配置，也可以从框架和配置追到代码

`generated/governance_tree.json` 则把这种依赖关系展开成树：

- 项目根
- framework 节点
- product_spec section 节点
- implementation_config section 节点
- code symbol 节点
- evidence artifact 节点

所以它证明的不只是“有文件”，而是“有结构化依赖图”。

## 6. 为什么我判断这条链是成立的

我判断“对得上”，核心不是因为概念上看起来合理，而是因为下面四件事同时成立：

### 6.1 单向收敛成立

没有看到运行时代码反向改写 framework 边界、规则或 `product_spec` 真相。

### 6.2 编译入口唯一

`generation_manifest.json` 明确记录当前唯一生成入口是：

- `project_runtime.knowledge_base.materialize_knowledge_base_project`

这意味着知识库主链不是多套生成器并存。

### 6.3 高风险行为都有治理绑定

页面路由、UI contract、backend surface、API contract、回答行为都在治理清单里有 `governed_symbol` 绑定和 upstream refs。

### 6.4 证据产物与代码都能互相解释

- `implementation_bundle.py` 记录了当前产品规格、结构合同、验证结果与生成物位置
- `governance_manifest.json` 记录了代码符号与上游结构来源
- `governance_tree.json` 把这种关系做成可遍历的依赖树

这说明：

- 框架不是悬空文档
- 代码也不是脱离框架的第二套真相

## 7. 当前仍然存在的边界说明

这条链已经成立，但还有两个需要明确说清楚的边界：

- 当前知识库模板是单模板主链，`knowledge_base_workbench` 之外的模板并没有进入同等成熟度的主运行链。
- 当前 release 交付是仓库侧版本说明 + 本地打包产物，不等于已经在 GitHub Releases 上完成公开发布页托管。

这两个边界不影响“从框架到代码的链路是否对得上”，但影响“它是否已经变成公共发行体系”。

## 8. 一句话总结

当前 `knowledge_base_basic` 不是“拿框架文档当说明书，再手写一个 demo”。

它已经是：

> 用 framework markdown 定义结构空间，用 `product_spec.toml` 固定产品真相，用 `implementation_config.toml` 固定实现路径，再把结果编译成 generated artifacts、runtime code 和 governance evidence 的一条真实编译链。
