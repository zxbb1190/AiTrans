# 从框架到代码映射与反查覆盖说明

## 1. 先给结论

当前 `shelf` 的知识库主链，已经不是“框架写一套、代码手写另一套”的关系。

对 `projects/knowledge_base_basic/` 这条链来说，仓库现在已经能做到：

- 从 `Framework -> Product Spec -> Implementation Config -> Code -> Evidence` 单向收敛到具体代码。
- 对一组高风险代码符号，反向检查“这段代码是否仍然对应既定框架与配置”。
- 对 `implementation_config.toml` 的叶子字段，检查它们是否真的进入下游行为，而不是变成死配置。
- 对 canonical generated artifacts，检查是否仍然由当前框架、产品和实现配置重新物化得到。

但也必须明确：

> **它还不能“100% 覆盖仓库里的全部代码”。**

当前做到的是：

- 对高风险结构做强覆盖
- 对编译产物做强一致性覆盖
- 对实现配置做字段级生效性覆盖

还没有做到的是：

- 对所有低风险辅助函数逐行建立 framework 反查
- 对所有运行时细节都做完整语义证明

所以更准确的判断是：

> **现在已经能覆盖“主链关键行为”，但还没有覆盖“全部代码”。**

---

## 2. 用一个具体例子说明：`/api/knowledge/product-spec` 这条路由是怎么来的

这条路由是一个很适合讲清楚全链路的例子，因为它同时经过了：

- framework 规则
- product spec
- implementation config
- 编译器
- runtime code
- governance evidence
- 反向校验

### 2.1 它最终落到哪段代码

最终运行时路由定义在 [app.py](/home/xue/code/shelf/src/knowledge_base_runtime/app.py)。

核心代码是：

```python
transport = _require_backend_transport(resolved)

@app.get(transport["product_spec_endpoint"])
def product_spec() -> dict[str, object]:
    return resolved.to_product_spec_dict()
```

这说明最终生效的路径不是写死在 `@app.get("/xxx")` 里，而是从编译后的 `backend_spec.transport.product_spec_endpoint` 取出来的。

---

## 3. 这条路由往上是怎么来的

### 3.1 Framework：先规定“必须存在稳定页面/接口路由”

对这条运行时路由，治理系统当前把它归到符号 `kb.runtime.page_routes`。

它在治理定义里的直接上游是：

- frontend L2 标准模块的 `R1 / R2 / R3`
- `product_spec.toml` 的 `route` section
- `implementation_config.toml` 的 `evidence` section

对应定义在 [governance.py](/home/xue/code/shelf/src/project_runtime/governance.py) 的 `_definitions()` 里，`symbol_id="kb.runtime.page_routes"` 这一项会显式声明：

- 它属于哪个 `governed_symbol`
- 它允许绑定哪些代码位置
- 它的 expected evidence 怎么构造
- 它的 actual evidence 怎么从代码里提取
- 它依赖哪些 framework rule / product section / implementation section

也就是说，框架不是模糊地“影响了代码”，而是通过治理定义把“哪个代码符号受哪些上游约束”写死了。

### 3.2 Product Spec：先固定产品 API 大类边界

产品路径前缀来自 [product_spec.toml](/home/xue/code/shelf/projects/knowledge_base_basic/product_spec.toml) 的 `[route]`：

- `api_prefix = "/api/knowledge"`

这是产品真相。它回答的是：

- 这个产品的 API 挂在什么前缀下

它不回答：

- 这个 API 是否需要额外暴露一条产品真相接口
- 那条接口叫什么

因为后者已经进入实现层。

### 3.3 Implementation Config：再细化实现路径

实现层的具体接口名来自 [implementation_config.toml](/home/xue/code/shelf/projects/knowledge_base_basic/implementation_config.toml) 的 `[evidence]`：

- `product_spec_endpoint = "/api/knowledge/product-spec"`

这一步回答的是：

- 当前实现路径准备把产品真相暴露在什么接口下

它不重新定义 `api_prefix` 是什么，而是在产品已定义的前缀之下细化一个实现层出口。

### 3.4 编译器：把上下游收敛成编译后结构

编译发生在 [knowledge_base.py](/home/xue/code/shelf/src/project_runtime/knowledge_base.py)。

这里有两步关键收敛：

1. `_validate_implementation_config()`

它会检查：

- `evidence.product_spec_endpoint` 必须落在 `route.api_prefix` 下

也就是，如果你把实现配置改成了 `/other/product-spec`，这一步就直接失败，不会进入物化。

2. `_build_backend_spec()`

它会把实现层出口写进编译后的 backend spec：

```python
"transport": {
    "mode": project.implementation.backend.transport,
    "api_prefix": project.route.api_prefix,
    "product_spec_endpoint": project.implementation.evidence.product_spec_endpoint,
}
```

这意味着：

- runtime code 不再自己猜配置
- 它只消费编译后结构

### 3.5 Runtime：只消费编译结果，不回推产品真相

运行时代码 [app.py](/home/xue/code/shelf/src/knowledge_base_runtime/app.py) 先调用 `_require_backend_transport()`，再用：

- `transport["product_spec_endpoint"]`

去挂真正的 FastAPI 路由。

因此这条链是：

`framework rule -> product route -> implementation evidence -> compiled backend_spec.transport -> runtime @app.get`

不是：

`runtime code 自己决定 route`

## 4. 代码改了以后，怎么反过来检查 framework / spec / config 有没有对应

这件事当前不是靠“人眼看 diff”，而是靠三层自动反查。

### 4.1 第一层：generated artifacts 反查

[validate_strict_mapping.py](/home/xue/code/shelf/scripts/validate_strict_mapping.py) 里的 `validate_project_generation_discipline()` 会做这件事：

- 重新调用 `materialize_knowledge_base_project()`
- 在临时目录重物化一套 generated artifacts
- 和当前 `projects/knowledge_base_basic/generated/*` 做字节级对比

因此如果你：

- 手改了 `generated/implementation_bundle.py`
- 手改了 `generated/generation_manifest.json`
- 手改了 `generated/governance_manifest.json`

都会直接被认为是 stale / manually edited。

这层保证的是：

> **generated evidence 不能脱离 framework / product spec / implementation config 单独漂移。**

### 4.2 第二层：governed symbol 反查

治理反查的核心是：

- [governance.py](/home/xue/code/shelf/src/project_runtime/governance.py) 里的 `_definitions()`
- [validate_strict_mapping.py](/home/xue/code/shelf/scripts/validate_strict_mapping.py) 里的 `validate_project_governance()`

对 `kb.runtime.page_routes` 这个例子，治理系统会分别构造：

1. expected evidence

由 `_expected_runtime_page_routes()` 生成，来源是：

- `product_spec.route`
- `implementation.evidence.product_spec_endpoint`

2. actual evidence

由 `_actual_runtime_page_routes()` 生成，它真的会：

- 构造 FastAPI app
- 读取 runtime 里实际注册出来的路由

然后 `compare_project_to_tree()` 比较 expected 和 actual。

如果你在 [app.py](/home/xue/code/shelf/src/knowledge_base_runtime/app.py) 里把：

```python
@app.get(transport["product_spec_endpoint"])
```

改成：

```python
@app.get("/api/knowledge/spec")
```

而没有同步修改上游配置与治理证据，那么：

- expected 仍然是 `/api/knowledge/product-spec`
- actual 会变成 `/api/knowledge/spec`

治理比较就会失败。

这层保证的是：

> **关键代码符号不能脱离上游 framework / spec / config 合同私自漂移。**

### 4.3 第三层：实现配置生效性反查

这轮新增的“配置即功能”校验在 [validate_strict_mapping.py](/home/xue/code/shelf/scripts/validate_strict_mapping.py) 的 `validate_implementation_config_effects()`。

它会做几件事：

1. 读取 `implementation_config.toml` 的所有叶子字段
2. 调用 [knowledge_base.py](/home/xue/code/shelf/src/project_runtime/knowledge_base.py) 里的 `build_implementation_effect_manifest()`
3. 检查每个字段有没有声明下游目标
4. 检查目标路径是否真的存在于 runtime bundle
5. 检查目标值是否真的反映当前配置值

例如：

- `evidence.product_spec_endpoint`
  - 必须映射到 `backend_spec.transport.product_spec_endpoint`
- `backend.retrieval_strategy`
  - 必须映射到 `backend_spec.retrieval.strategy`
- `frontend.style_profile`
  - 必须映射到 `ui_spec.implementation.style_profile`

这层保证的是：

> **配置字段不能只是写在 TOML 里或抄进 bundle 里，它必须真正进入下游行为或编译结果。**

## 5. 新增代码时，系统如何拦住“脱框架”的高风险实现

当前对高风险代码还有一个额外拦截：

[governance.py](/home/xue/code/shelf/src/project_runtime/governance.py) 的 `find_unbound_high_risk_structures()` 会扫描：

- `src/knowledge_base_runtime/app.py`
- `src/knowledge_base_runtime/backend.py`

它当前特别盯的是：

- `build_knowledge_base_runtime_app()` 里的路由函数
- `build_knowledge_base_router()` 里的路由函数

如果你在这些 builder 里新增了一个新的 `@app.get()` / `@router.post()` 路由，但没有加对应的：

```python
# @governed_symbol ...
```

治理快照构建会直接失败。

这意味着至少在知识库主链最敏感的“页面入口 / API 入口”这类高风险点上，仓库不是放任你新增未治理代码的。

## 6. 现在到底能不能“全部覆盖代码”

答案必须分开说。

### 6.1 能强覆盖的部分

当前能强覆盖的，是以下几类：

1. canonical generated artifacts

- `framework_ir.json`
- `product_spec.json`
- `implementation_bundle.py`
- `generation_manifest.json`
- `governance_manifest.json`
- `governance_tree.json`

这几份文件都要通过重物化比对。

2. 8 个高风险 governed symbols

当前治理清单里已经强绑定了：

- `kb.runtime.page_routes`
- `kb.frontend.surface_contract`
- `kb.workbench.surface_contract`
- `kb.ui.surface_spec`
- `kb.backend.surface_spec`
- `kb.api.library_contracts`
- `kb.api.chat_contract`
- `kb.answer.behavior`

这些符号都有：

- expected builder
- actual extractor
- upstream refs
- required bindings

3. `implementation_config.toml` 的叶子字段

当前知识库模板下，以下字段都会被自动检查是否有下游效果：

- `frontend.renderer`
- `frontend.style_profile`
- `frontend.script_profile`
- `backend.renderer`
- `backend.transport`
- `backend.retrieval_strategy`
- `evidence.product_spec_endpoint`
- `[artifacts]` 下所有产物名

### 6.2 只能间接覆盖的部分

还有一些代码，目前只是间接被覆盖：

- [frontend_style.py](/home/xue/code/shelf/src/knowledge_base_runtime/frontend_style.py)
- [frontend_script.py](/home/xue/code/shelf/src/knowledge_base_runtime/frontend_script.py)
- [frontend.py](/home/xue/code/shelf/src/knowledge_base_runtime/frontend.py) 里的页面拼装细节

它们现在的覆盖方式主要是：

- 通过 `ui_spec` / implementation selection 被约束
- 通过 governed symbol 所抽取出来的 contract 被约束
- 通过运行时最终路由和行为结果被约束

但这不等于：

- 每一段 CSS
- 每一段 DOM 拼装
- 每一个 JS helper

都已经有单独的 framework 反查定义。

### 6.3 目前还不能全覆盖的部分

当前没有做到全覆盖的，主要是：

1. 低风险辅助函数

例如：

- 字符串拼接 helper
- 一些内部转换函数
- 一般性的工具函数

只要它们不属于治理定义中的高风险符号，就没有逐个 framework 反查。

2. 未被抽成 expected/actual contract 的内部算法细节

例如 [backend.py](/home/xue/code/shelf/src/knowledge_base_runtime/backend.py) 里的 `_rank_sections_stub()`：

- 如果你改的是排序细节
- 但改动没有超出当前 answer behavior / backend surface spec 的抽取范围

那它不一定会立刻被 framework 反查发现。

3. legacy / example / sample 代码

当前主治理链重点覆盖的是知识库主线，不是仓库所有历史样本。

## 7. 所以，这套机制的真实能力边界是什么

更准确地说，当前仓库做到的是：

> **对“项目主链上的关键结构代码”建立 framework 反查，而不是对“仓库所有代码”建立逐行映射。**

这套机制已经很强，但它不是魔法。

它现在更像：

- 对关键页面入口、API 契约、编译 surface、回答行为做强治理
- 对 generated evidence 做强一致性校验
- 对实现配置做字段级生效性校验

而不是：

- 对任何 Python/JS/CSS 改动都自动找到唯一的 framework 依据

---

## 8. 如果要继续逼近“全部覆盖代码”，还缺什么

如果要再往前推一层，至少还要补三类东西：

### 8.1 把更多高风险 helper 提升成 governed symbol

例如：

- 前端页面拼装器的更多关键函数
- 后端检索与排序行为的更多关键阶段
- 配置解释器里的更多关键 builder

### 8.2 扩大 expected/actual contract 的抽取面

当前很多检查只抽取了“表面合同”。

如果要更强，需要把这些也纳入：

- 更多 runtime state 约束
- 更多排序/检索行为约束
- 更多前端可见结构约束

### 8.3 引入更细粒度的行为测试证据

现在主要是：

- 结构比对
- route 比对
- contract 比对
- manifest/tree closure 比对

如果要更靠近“全部覆盖”，还需要：

- 更明确的行为用例
- 更细的 end-to-end 断言
- 把这些行为证据也纳入 governance

---

## 9. 最后的判断

如果问题是：

> 现在能不能从框架一路落到具体代码？

答案是：

> **能，而且已经是可执行、可物化、可反查的。**

如果问题是：

> 现在代码改动后，能不能反过来检查 framework / product spec / implementation config 有没有对应？

答案是：

> **对知识库主链上的高风险结构，能。**

如果问题是：

> 现在是否真的已经覆盖全部代码？

答案是：

> **还没有。当前覆盖的是主链关键结构，不是整个仓库的全部实现细节。**

所以最准确的一句话总结是：

> **当前 `shelf` 已经做到了“关键结构代码可从框架反查”，但还没有做到“仓库全部代码完全被框架覆盖”。**
