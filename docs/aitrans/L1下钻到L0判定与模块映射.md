# framework L1 下钻到 L0 判定与模块映射

@framework

## 1. 判定规则

- `继续保留在 L0`：同一结构单元在多个 `L1` 模块中被稳定复用，且继续下钻只会退化为实现对象、规则表或函数级细节。
- `共享吸收到 L0`：多个 `L1` 模块沿同一拆分轴重复引用相同骨架时，应优先折叠为共享 `L0` 叶子。
- `停在 L1`：若继续拆分不会产生新的独立结构选择，只会生成 Handler / Builder / Validator / UI 子组件等实现细节，则不再新增更细模块。
- `物理合并优先`：当若干 `L0` 叶子只在单一 `L1` 中以固定顺序、固定成对方式出现，应直接物理合并，而不是保留多个伪差异叶子。

## 2. 当前收敛结果

- 当前 `aitrans` 结构为 `L3=1 / L2=5 / L1=23 / L0=36`，且 `L0-M0 ~ L0-M35`、`L1-M0 ~ L1-M22` 已连续编号，无空洞。
- `L1-M5[交互会话管理器]` 已吸收旧截图会话与旧展示会话的同构骨架。
- `L1-M11[提供方执行适配器]` 已吸收旧 OCR Provider 与旧翻译 Provider 的镜像适配骨架。
- `L0-M23[面板布局约束器]` 已吸收旧布局定位叶子与旧避障叶子。
- `L0-M30[链路诊断收敛器]` 已吸收旧链路上下文叶子与旧诊断快照叶子。
- `L0-M35[发布更新治理器]` 已吸收旧产物版本叶子与旧更新策略叶子。

## 4. 已完成的关键物理合并

- `旧 L1-M5 + 旧 L1-M15 -> 当前 L1-M5[交互会话管理器]`
- `旧 L1-M11 + 旧 L1-M12 -> 当前 L1-M11[提供方执行适配器]`
- `旧 L0-M32 + 旧 L0-M36 -> 当前 L0-M23[面板布局约束器]`
- `旧 L0-M48 + 旧 L0-M49 -> 当前 L0-M30[链路诊断收敛器]`
- `旧 L0-M57 + 旧 L0-M59 -> 当前 L0-M35[发布更新治理器]`

## 5. 当前仍保留分离的原因

- `L1-M7 / L1-M8 / L1-M9` 虽同属截图链路，但分别对应采集、坐标换算、结果统一，变化来源不同。
- `L1-M14 / L1-M15 / L1-M16 / L1-M17` 虽同属展示链路，但分别对应内容渲染、动作处理、空间布局、状态驱动，仍有真实结构差异。
- `L1-M20 / L1-M21 / L1-M22` 都属于治理域，但分别承载观测、安全、发布，当前再强并会把治理边界混掉。

## 6. 当前建议

1. 先停止继续做模块级物理压缩，避免为压缩而压缩。
2. 若继续优化，优先统一错误对象模型、审计字段和输出契约，而不是再减少模块个数。
3. 新一轮合并只应在再次出现“固定成对消费、且不提供新结构选择”的叶子时再做。
