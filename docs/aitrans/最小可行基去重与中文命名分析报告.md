# framework 最小可行基去重与中文命名分析报告

## 1. 结论摘要

- 当前 `framework/aitrans` 共 `65` 个模块文档，其中 `L0=36 / L1=23 / L2=5 / L3=1`。
- 当前全目录共定义 `114` 个“最小可行基”条目。
- 经过连续几轮收敛后，最明显的模块级镜像已经被压掉，当前不存在新的高确定性物理合并目标。
- 当前更值得做的是“契约统一”和“错误表达一致性”，不是继续机械减少模块数。

## 2. 已完成的高价值收敛

| 收敛动作 | 结果 | 收益 |
| --- | --- | --- |
| 旧截图会话模块 + 旧展示会话模块 | 合并为 `L1-M5` 交互会话管理器 | 去掉双份会话骨架，截图与展示共用同一交互会话面 |
| 旧 OCR Provider 模块 + 旧翻译 Provider 模块 | 合并为 `L1-M11` 提供方执行适配器 | 去掉 OCR / TRANS 镜像适配器，`L2-M2` 的执行面收成单模块 |
| 旧布局定位叶子 + 旧避障叶子 | 合并为 `L0-M23` 面板布局约束器 | 去掉固定成对出现的布局叶子，展示布局面继续保持清晰 |
| 旧链路上下文叶子 + 旧诊断快照叶子 | 合并为 `L0-M30` 链路诊断收敛器 | `L1-M20` 从三基收成两基，链路与诊断回路更紧 |
| 旧产物版本叶子 + 旧更新策略叶子 | 合并为 `L0-M35` 发布更新治理器 | `L1-M22` 从三基收成两基，发布链路边界更完整 |
| 历史上的窗口、选区、坐标、回退、展示、IPC、观测、安全、构建等旧五连模板 | 压缩为当前 `36` 个 `L0` 叶子 | 去掉大量只换名字不换结构的叶子模块 |

## 3. 当前仍需关注的重叠点

### 3.1 错误 / 输出契约

涉及模块：

- `L1-M12` 回退策略控制器
- `L1-M13` 识别翻译结果装配器
- `L1-M18` 进程间通信桥接网关
- `L1-M20` 可观测性中心

判断：

- 错误分流、降级结果、标准输出、审计追踪仍在多条链路中复用。
- 这说明下一步应统一错误对象与输出契约，而不是继续把模块压小。

### 3.2 治理域共享字段

涉及模块：

- `L1-M18` 进程间通信桥接网关
- `L1-M19` 配置中心
- `L1-M20` 可观测性中心
- `L1-M21` 安全策略守卫
- `L1-M22` 构建发布管理器

判断：

- 这些模块都使用审计、追踪、环境、权限等共享字段。
- 现在的优化方向应是统一字段模型和契约，而不是把通信、安全、观测、发布强行合并。

## 4. 不建议继续强并的边界

- `L1-M7` 屏幕帧获取器、`L1-M8` 坐标映射引擎、`L1-M9` 截图结果标准化器：分别承接采集、坐标换算、结果统一，变化原因不同。
- `L1-M14` 结果视图渲染器、`L1-M15` 交互动作控制器、`L1-M16` 悬浮面板布局管理器、`L1-M17` 展示状态机：分别承接内容、动作、空间、状态，强并会导致展示链职责混叠。
- `L1-M18` 进程间通信桥接网关、`L1-M21` 安全策略守卫、`L1-M22` 构建发布管理器：虽然都在治理域，但承载的是通信、安全、发布三类不同结构。
- `L0-M0 ~ L0-M9` 通用原子：虽然文档模板相似，但它们是跨域稳定复用的更细通用结构，不应为追求数字更小而硬并。

## 6. 当前建议

1. 物理模块收敛建议先停在当前状态。
2. 下一步若继续优化，优先统一错误对象、审计字段、诊断字段和输出契约。
3. 只有再次出现“固定成对消费，且不提供新结构选择”的叶子，才值得继续做模块级合并。
