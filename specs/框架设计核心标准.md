# 框架设计标准（核心）

## 0. 定位
本文件定义框架设计核心方法，不包含工程执行细节。

核心范围仅包含五项：
1. 能力声明
2. 边界定义
3. 最小可行基
4. 基组合原则
5. 验证

补充定位：
- 结构是框架的第一性对象；能力不是结构，能力是基在边界条件下经由组合规则稳定导出的结果。
- 一切最小可行基都必须是结构，不能把动作、状态、目标或功能清单直接当作基。
- 框架设计必须先从更基础、更通用的结构开始，再向更高层的领域结构和场景结构组合。
- 若某类框架天然存在更细的通用原子，例如前端中的按钮、输入框、文本块、列表、树、标签与气泡，则这些原子应优先落在最低可复用结构层，不应先被合并成过粗的“组件族”再作为起点。
- 通用框架既可以是基础通用框架，也可以是领域通用框架；判断标准是“是否跨实例稳定复用”，而不是“是否完全无领域语义”。
- 更高层框架可以建立在更基础的通用框架之上，但依赖方向必须单向：只能从更基础结构流向更高抽象结构，不得反向污染。
- 框架首先定义基、边界与规则，不首先定义功能清单、实例值或实现细节。
- 设计意图或初始目标可以记录，但不属于框架核心必选项。
- 层级模型不是固定模板；是否分层、分多少层、每层语义为何，均由具体框架自行声明。
- 当框架被用作 AI 编程输入语言时，框架文档仍然只定义结构标准；生成代码、测试与运行产物属于实现与证据层，不反向定义框架本体。
- 当框架被用作项目实例化源时，实例层只能通过 `projects/<project_id>/instance.toml` 固定边界内允许变化的值；`instance.toml` 不得反向修改 `B*`、`R*` 或框架边界定义本身。
- 当框架被用作项目代码生成源时，项目实例行为只能由 `framework/*.md` 与 `projects/<project_id>/instance.toml` 驱动；`projects/<project_id>/generated/*` 属于生成产物，不得作为人工编辑源。
- 当框架被用作项目实例化源时，边界定义优先保持大类语义；`instance.toml` 应以同名大类 section 承载实例值，小类字段只在所属大类 section 内展开。

## 1. 能力声明（Capability Statement）
定义框架负责导出的能力范围与非能力项。

要求：
- 能力声明必须回答“该框架能稳定导出什么能力”。
- 能力声明必须回答“该框架不负责什么”。
- 能力不是目标项，不是功能清单，也不是实例配置。
- 同一组基与规则在不同边界下可以导出不同能力集合；能力应以结构导出关系表达。

## 2. 边界定义（Boundary）
定义框架适用范围与限制条件。

要求：
- 每个边界项都可测量
- 每个边界项都可校验
- 边界项需直接服务于能力成立条件、适用域或验证条件
- 边界用于约束基的展开空间，不用于定义具体实例值
- 若框架被编译为实例配置，边界优先保持大类抽象；实例配置按同名大类 section 组织，小类字段不得脱离所属边界游离出现
- 若某个边界被开放到实例层，则该边界必须能稳定追溯到 `instance.toml` 中的主 section；边界引用与配置落点之间不得出现不可解释的中间杂项层
- 当基是 UI 基础结构时，边界至少应覆盖视觉语义边界与渲染绑定边界；前者约束形状、边框、色调、密度等语义外形，后者约束宿主元素、属性绑定、事件出口与框架交互方式。

举例说明：
- 可以抽象 `Button` 结构能力，不应在基定义中固化“红色按钮”这类实例。
- 颜色、尺寸、主题等实例差异应通过边界参数表达。

## 3. 最小可行基（Bases）
定义最小可行基（Minimal Viable Base）。

要求：
- 模块是文件级单元：一个 `framework/<domain>/Lx-Mn-*.md` 文件对应一个模块。
- 当同层存在多个模块文件时，文件名必须使用 `Lx-Mn-*.md` 表示模块编号（例如 `L1-M0-...md`）。
- 模块别名（若需要）是文件级信息，不得绑定到单个 `B*`。
- 每个基必须具备 `B{n}` 标识
- 每个基必须能映射为 `L{X}.M{m}.B{n}` 的模块内结构单元
- 基是最小、可解耦、可复用、可组合、跨实例稳定的叶子结构单元，不是模块单元
- 基的本质在于叶子性，而不在于是否位于 `L0`
- `B*` 必须是抽象、可复用、跨实例稳定的结构单元
- 最小可行基应尽量收敛自由度，在满足当前层生成要求的前提下，选择自由度最少且最合适的结构
- 基的命名与定义必须优先描述组织形式，不得用动作词、状态词或目标词替代结构定义
- 功能必须由结构导出，不得把功能清单直接当作基定义
- `B*` 格式应为 `B* 名称：<结构定义或 Lx.My[...] 或 framework.Lx.My[...]>。来源：\`...\``；若引用本框架更低层模块或外部更基础通用框架，必须直接将模块引用内联写在主句中，不得追加 `上游模块：...` 字段，基级细节由组合原则约束。
- 若框架声明层级，则最低被引用层上的叶子基必须保持自足；根层 `L0` 的 `B*` 在本框架内部必须自足，不得再引用本框架其它模块。
- `L0` 若要显式依赖外部更基础通用框架，只允许使用 `framework.L0.Mn[...]` 或 `framework.L1.Mn[...]` 这类基础引用语法，不得借此绕开本框架后续层级定义。
- 若更高层领域框架建立在更基础的通用框架之上，应显式声明该依赖方向；更基础的通用结构不应反向依赖更高层领域结构。

## 4. 组合原则（Combination Principles）
定义基如何形成有效系统。

要求：
- 组合规则必须可表达为显式条件
- 规则间不可互相冲突
- 组合结果必须可判定“有效/无效”
- 组合规则必须说明参与基、组合方式、导出能力与边界绑定
- 规则应能区分合法组合、非法组合与不适用情形
- 若框架声明层级，则组合只允许发生在相邻可见层：`Lx -> L(x+1)`（下游向上游）
- 若框架声明层级，则禁止同层组合与跨层跳级组合
- 允许同一下层节点被多个相邻上层节点复用，但每个非叶子节点必须至少由一个相邻下层节点组成
- 从树结构看，基础结构落在叶子节点，其它节点都由下层节点逐层组成
- 若框架显式依赖外部更基础通用框架，该依赖必须只承担承接面，不得替代当前框架内部的结构定义责任

## 5. 验证（Verification）
定义如何判断框架结论是否成立。

要求：
- 必须有通过条件（pass criteria）
- 必须有失败原因（failure reasons）
- 必须有证据（evidence）
- 必须可自动验证层级与组合合法性
- 必须可区分“结构定义问题”与“实例配置问题”
- 必须验证“声明的能力是否真的由这些基在边界条件下经由规则导出”
- 当边界被实例化为配置 section 时，验证必须覆盖“边界 -> 主 section”的稳定映射，并能定位到对应配置落点
- 当框架被用作生成式实现输入时，验证应覆盖“框架 -> 实现”一致性，并将生成代码、测试结果与运行样例作为证据层产物
- 当框架被用作项目生成源时，验证还必须覆盖“实例目录内无手写项目代码”和“当前 generated 产物可由框架与实例配置重新生成且完全一致”
- 只有当框架本身是比较式或优化式框架时，才强制要求 `baseline/target`

## 6. 外部关联
本文件的外部关联由顶层规范树定义：
- `specs/规范总纲与树形结构.md`
- `specs/code/Python实现质量标准.md`
- 层级传导方向：`L0 -> L1 -> L2 -> L3`
- 模块框架文件命名约束：`framework/<module>/Lx-Mn-*.md`（层级由 `Lx-` 前缀判定）
- L0-L3 树节点仅承载框架标准文件；标准到实现的对应关系在 `mapping/mapping_registry.json` 的 `mappings` 中维护。
- 追溯、映射、变更传导和 lint 属于治理层能力，由顶层规范、映射注册与校验脚本处理，不属于本文件的最小定义。
