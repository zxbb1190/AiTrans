# 框架设计标准（核心）

## 0. 目标（可选）

框架文档可以在开头记录设计目标、问题入口、关注对象或期望导出的结果类型，也可以省略不写。

目标的作用是提供探索起点，而不是充当结构正确性的第一性依据。若后续结构推导揭示出更稳定、更一般或更具生成性的组织方式，应允许修正目标表述，而不是反向让结构服从初始目标。

## 0.1 定位
本文件定义框架设计核心方法，不包含工程执行细节。

核心范围分为两部分：
1. 结构定义
2. 五项设计方法：能力声明、边界定义、最小可行基、基组合原则、验证

本文件采用以下基本立场：
- 结构是框架的第一性对象。
- 框架不是某个项目的模板，而是 AI 编程时代的人和 AI 之间的共同结构语言。
- 通用框架既可以是基础通用框架，也可以是领域通用框架；判断标准是“是否跨实例稳定复用”，而不是“是否完全无领域语义”。
- 设计意图或初始目标可以记录，但不属于框架核心必选项；若结构推导揭示出更稳定、更一般或更具生成性的组织方式，应允许修正目标表述，而不是反向让结构服从初始目标。
- 层级模型不是固定模板；是否分层、分多少层、每层语义为何，均由具体框架自行声明。
- 本文件只定义框架设计的语义核心。

## 0.2 结构定义（Structure）
人们谈论一个事物时，往往最先看到的是它的功能：置物架能放东西，桥能承重，语言能表达思想，程序能处理输入，组织能完成任务，生物能生长和适应。

但功能不是凭空出现的。一个事物之所以能够稳定地、重复地生成某种结果，并不是因为“功能”这个词本身具有力量，而是因为其内部已经存在某种更基础的组织方式，使这种结果得以成立。这个更基础的东西，就是结构。

结构不是结果，不是表现，不是实现，也不是实例；它不是“发生了什么”，而是“为什么能够这样发生”，是事物内部各要素之间得以稳定成立、相互配合并持续生成结果的组织关系。

功能是结构的表现，变化是结构的变形，演化是结构的重组。

页面如此，程序如此，组织如此，语言如此，机械如此，生命亦如此。凡能生成结果者，背后必有结构。

因此，在本框架中，结构是最基础、最原始、也是最核心的概念。

如需进一步展开理解“什么是结构”以及“最小可行基如何参与生成更高层结构”，参见 [结构详解：从最小可行基到万物生成](./结构详解-从最小可行基到万物生成.md)。

## 1. 能力声明（Capability Statement）
定义框架负责导出的能力范围与非能力项。

要求：
- 能力声明必须回答“该框架能稳定导出什么能力”。
- 能力声明必须回答“该框架不负责什么”。
- 能力不是结构，能力是基在边界条件下经由组合规则稳定导出的结果。
- 同一组基与规则在不同边界下可以导出不同能力集合；能力应以结构导出关系表达。

## 2. 边界定义（Boundary）
定义框架适用范围与限制条件。

边界描述的不是“变化本身”，而是“结构成立时允许变化的维度、范围与成立条件”。

也就是说，边界回答的是：
- 什么可以变
- 变到哪里仍然算该结构的合法展开
- 超出哪里就不再属于当前结构问题域

因此，边界不是对 `B*` 的配套参数清单，也不是某个实例的具体取值；边界首先约束的是结构展开空间，而不是反过来从现有基定义里抽取说明项。

要求：
- 每个边界项都可测量
- 每个边界项都可校验
- 边界项需直接服务于能力成立条件、适用域或验证条件
- 边界用于约束基的展开空间，不用于定义具体实例值
- 边界应优先表达“允许变化的维度 + 合法范围/作用域/窗口/条件”，而不是对象名词清单或对某个 `B*` 的附属注释
- 边界不是 `B*` 本体，不得把结构单元、动作步骤或状态机节点直接当作边界命名
- 边界不是实例值；应表达为可测的合法区间、作用域、窗口、阈值或判定条件，而不是直接写死某个具体值

## 3. 最小可行基（Bases）
定义最小可行基（Minimal Viable Base）。

要求：
- 基必须能被稳定定位到某个模块内结构单元。
- 框架设计必须先从更基础、更通用的结构开始；若某类框架天然存在更细的通用原子，应优先把这些原子落在最低可复用结构层，而不是先合并成过粗的结构族
- 基是当前模块内最小、可解耦、可复用、可组合、跨实例稳定的结构单元
- 基的“最小”是相对于当前模块的最小可行表达边界，而不是要求所有 `B*` 在整个框架引用图里都是全局叶子
- 一切最小可行基都必须是结构，不能把动作、状态、目标或功能清单直接当作基
- `B*` 必须是抽象、可复用、跨实例稳定的结构单元
- 最小可行基应尽量收敛自由度，在满足当前层生成要求的前提下，选择自由度最少且最合适的结构
- 基的命名与定义必须优先描述组织形式，不得用动作词、状态词或目标词替代结构定义
- 功能必须由结构导出，不得把功能清单直接当作基定义
- 若多个模块只是重复同一拆分轴而不提供新的独立结构选择，则应主动合并模块、压缩层级与收敛 `B*` 数量，直到每个保留下来的模块和基都代表真实独立的结构差异
- 若框架声明层级，则更高层 `B*` 可以由更低层模块组合而成；未继续向下展开的最低被引用层基必须保持自足，且根层 `L0` 的 `B*` 在本框架内部必须自足，不得再引用本框架其它模块。
- 若框架显式依赖外部模块，则必须显式声明该依赖；外部依赖是否合法优先由引用图的单向性、存在性与无环性判断，被引模块的 `Lx` 标签仅作为总体规划参考，不得代替真实依赖审查。
- 记正向能力集合为 `C+`、最小可行基集合为 `B`。最小可行基层的“基 -> 能力”对应关系记为 `A: B -> 𝒫(C+)`，其中 `A(b)` 表示基 `b` 在基层直接对应的能力子集。
- `A(b)` 可以包含一个或多个能力；若某个基当前只承担组合支撑角色而不直接承接正向能力，则其 `A(b)` 允许为空，但该基仍必须进入至少一条合法组合规则。
- 对任一正向能力 `c ∈ C+`，必须存在唯一一个基 `b ∈ B` 使 `c ∈ A(b)`；也就是能力到基的关系 `g: C+ -> B` 必须是单值函数。
- 因此，能力在最小可行基层只能唯一归属到一个基；一个基则可以对应多个能力，或仅作为组合支撑基参与后续导出。

## 4. 组合原则（Combination Principles）
定义基如何形成有效系统。

要求：
- 显式模块引用关系是框架层次判断的第一性事实；`L0/L1/L2/...` 只是当前框架内部的局部投影、阅读分组和规划标注，不能反向覆盖真实依赖方向
- 组合原则只针对已定义的最小可行基进行选择、排序、约束与闭合，不应在组合阶段产生新的基；若需要新的独立结构单元，必须回到“最小可行基”层显式定义
- 组合规则必须可表达为显式条件
- 规则间不可互相冲突
- 组合结果必须可判定“有效/无效”
- 组合规则必须说明参与基、组合方式、导出能力与边界绑定
- 规则应能区分合法组合、非法组合与不适用情形
- 组合原则的作用，是把最小可行基潜在的无序排列组合空间，从无限、发散、不可判定的状态，收敛为有限、可解释、可验证的合法组合集合
- 组合必须沿“更基础结构 -> 更高抽象结构”的单向依赖展开，不得反向污染
- 若框架声明层级，则本框架内部的显式引用必须只指向更低的局部层；局部层差是否为 `1` 属于规划选择，不再单独作为硬约束
- 若框架声明层级，则禁止同层引用、反向引用与任何循环依赖
- 允许同一下层节点被多个更高节点复用，但每个组合节点必须至少由一个更低层节点组成
- 从树结构看，未继续向下展开的基础结构落在叶子节点，其它高层结构可由下层节点逐层组成
- 若更高层领域框架建立在更基础的通用框架之上，应显式声明该依赖方向；更基础的通用结构不应反向依赖更高层领域结构
- 若框架显式依赖外部更基础通用框架，该依赖必须只承担承接面，不得替代当前框架内部的结构定义责任
- 记组合原则集合为 `R`。组合原则对基的参与关系记为 `P: R -> 𝒫(B) \\ {∅}`；`P(r)` 表示规则 `r` 使用的基集合，因此一个组合原则可以使用一个或多个基。
- 组合原则对能力的导出关系记为 `O: R -> 𝒫(C+) \\ {∅}`；`O(r)` 表示规则 `r` 导出的能力集合，因此一个组合原则可以导出一个或多个能力。
- 对每个正向能力 `c ∈ C+`，至少必须存在一条合法导出链 `g(c) -> r -> c`，即至少存在一个组合原则 `r`，使得 `g(c) ∈ P(r)` 且 `c ∈ O(r)`。

## 5. 验证（Verification）
定义如何判断框架结论是否成立。

要求：
- 必须有通过条件（pass criteria）
- 必须有失败原因（failure reasons）
- 必须有证据（evidence）
- 必须可自动验证层级与组合合法性
- 必须可区分“结构定义问题”与“实例配置问题”
- 必须验证“声明的能力是否真的由这些基在边界条件下经由规则导出”
- 当边界被实例化为配置 section 时，验证必须覆盖“边界 -> 主 section”的稳定映射，并能定位到对应配置落点
- 当框架被用作生成式实现输入时，验证应覆盖“框架 -> 实现”一致性，并将生成代码、测试结果与运行样例作为证据层产物
- 当框架被用作项目生成源时，验证还必须覆盖“实例目录内无手写项目代码”和“当前 generated 产物可由框架与实例配置重新生成且完全一致”
- 只有当框架本身是比较式或优化式框架时，才强制要求 `baseline/target`

## 6. AI 编程分层职责（强制）

### 6.1 Framework 的职责边界

- `Framework` 负责定义共同结构、公理、能力范围、边界、组合规则、验证方式与合法展开空间。
- `Framework` 首先定义基、边界与规则，不首先定义功能清单、实例值或实现细节。
- `Framework` 只回答“什么结构成立、什么组合合法、哪些边界允许变化”，不直接回答某个具体产品最终长什么样。
- `Framework` 不得固化项目专属品牌、实例文案、主题值、知识内容、技术栈、库名、文件结构、类名、函数名或运行时实现技巧。
- `Framework` 可以定义默认规则、派生约束与合法值域，但这些规则必须保持跨实例稳定复用。
- 当框架被用作 AI 编程输入语言时，框架文档仍然只定义结构标准；生成代码、测试与运行产物属于实现与证据层，不反向定义框架本体。

### 6.2 Product Spec 的职责边界

- `Product Spec` 负责定义某一具体产品最终是什么，是该产品的语言无关、实现无关真相层。
- `Product Spec` 应表达页面与承载结构、交互语义、视觉语义、内容语义、业务对象、返回回路、外部契约与产品不变量。
- `Product Spec` 可以同时具备人可读表述和机器可读 canonical 结构，但两者必须表达同一产品真相。
- 以 `projects/<project_id>/product_spec.toml` 承载时，默认使用中文注释，并在文件头与每个主 section 前提供足以支撑产品讨论的详细说明；注释应解释该段产品真相、讨论重点及与 `Framework` / `Implementation Config` 的分界。
- `Product Spec` 不得泄漏技术栈、宿主框架、包管理、类名、函数名、文件路径、CSS 选择器、DOM 细节或其它仅属于某种实现方式的结构。

### 6.3 Implementation Config 的职责边界

- `Implementation Config` 负责把 `Product Spec` 细化为某一种实现路径、技术栈、运行时组织、生成策略与部署约束。
- `Implementation Config` 可以选择语言、框架、存储方案、接口承载方式、路由适配方式、代码布局与生成目标。
- `Implementation Config` 只能细化 `Product Spec` 尚未固化的实现维度，不得改写产品可观察真相，不得新增违反框架边界的结构。
- `Implementation Config` 必须保持可替换性：同一 `Product Spec` 应允许映射到多个实现配置，而不改变产品真相层。
- 以 `projects/<project_id>/implementation_config.toml` 承载时，默认使用中文注释，并在文件头与每个主 section 前说明当前细化的是哪类实现决策、如何服务既定 `Product Spec`、哪些产品真相不得由该层改写。

### 6.4 层间映射与泄漏约束

- 允许的映射：
  - `Framework -> Product Spec`：在框架边界内选择合法变体、补齐实例真相并导出具体产品不变量。
  - `Product Spec -> Implementation Config`：在不改变产品真相的前提下选择实现路径、技术栈与运行时组织。
  - `Implementation Config -> Code`：以生成或手写方式实现代码。
  - `Code -> Evidence`：生成测试结果、运行样例、构建产物与验证证据。
- 禁止的泄漏：
  - `Product Spec` 反向修改 `Framework` 的 `B*`、`R*`、边界定义或合法值域。
  - `Implementation Config` 反向改写 `Product Spec` 已确定的页面、交互、视觉语义、返回语义、内容语义或外部契约。
  - `Code` 成为产品真相、框架定义或实现配置的第一性事实来源。
  - `Evidence` 被当作人工编辑源反向定义 `Framework`、`Product Spec` 或 `Implementation Config`。
- 当框架被用作项目代码生成源时，项目行为只能由 `framework/*.md`、`projects/<project_id>/product_spec.toml` 与 `projects/<project_id>/implementation_config.toml` 驱动；`projects/<project_id>/generated/*` 属于生成产物，不得作为人工编辑源。
- 当前若仓库尚未把 `Product Spec` 与 `Implementation Config` 拆成独立文件，则临时合并承载文件必须在字段设计与验证上保持两层可分离，不得把实现细节直接伪装成产品真相。

## 7. 外部关联
本文件的外部关联由顶层规范树定义：
- `specs/规范总纲与树形结构.md`
- `specs/框架文档Lint标准.md`
- `specs/code/Python实现质量标准.md`
- 层级传导方向：`L0 -> L1 -> L2 -> L3`
- L0-L3 树节点仅承载框架标准文件；标准到实现的对应关系在 `mapping/mapping_registry.json` 的 `mappings` 中维护。
- 当框架边界被开放到产品层时，其关联落点是 `projects/<project_id>/product_spec.toml` 的同名大类 section；边界不应被拆散到不可追溯的杂项层。
- 框架文档表达、作者入口与 lint 约束由 `specs/框架文档Lint标准.md` 承接；本文件不重复定义文档协议。
- 追溯、映射、变更传导和 lint 属于治理层能力，由顶层规范、映射注册与校验脚本处理，不属于本文件的最小定义。
