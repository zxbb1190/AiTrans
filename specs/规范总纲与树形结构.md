# 规范总纲与树形结构（L0）

## 1. 目的
本文件是仓库规范总纲，定义标准树、模块分层目录约束和变更传导规则。

## 2. 标准树（强制）

标准文件目录采用“职责分离”：
- `specs/`：承载 L0/L1 抽象规范
- `framework/<module>/Lx-Mn-*.md`：承载领域模块分层标准
- `mapping/`：承载机器可读映射注册（L3）
- `projects/<project_id>/product_spec.toml`：承载项目产品规格（`Product Spec` 层，不属于 L0-L3 标准树）
- `projects/<project_id>/implementation_config.toml`：承载项目实现细化配置（`Implementation Config` 层，不属于 L0-L3 标准树）
- `projects/<project_id>/generated/`：承载由框架文档、产品规格与实现配置物化出的项目产物，属于证据层，不是人工编辑源

## 3. AI 编程分层语义（强制）

- 框架不是某个项目的模板，而是 AI 编程时代的人和 AI 之间的共同结构语言。
- 仓库主分层采用 `Framework -> Product Spec -> Implementation Config -> Code -> Evidence` 的单向收敛模型。
- `Framework` 层负责定义共同结构、公理、边界、组合规则与验证，不直接承载某个具体项目的品牌、文案、主题值、技术栈或实现细节。
- `specs/框架文档Lint标准.md` 负责定义框架标准文档的表达协议、作者入口与 lint 合同；它服务 `Framework` 层，但不改写框架语义。
- `Product Spec` 层负责定义某一具体产品最终是什么，必须保持对实现语言、技术栈与运行时组织方式无关。
- `Implementation Config` 层负责把 `Product Spec` 细化为某一种实现路径、技术栈、运行时与生成方式，不得改写 `Product Spec` 已确定的产品真相。
- `projects/<project_id>/product_spec.toml` 与 `projects/<project_id>/implementation_config.toml` 作为人与 AI 协作讨论入口，默认使用中文注释，并优先写清职责、边界与讨论重点。
- 当这类 TOML 文件篇幅可控时，应采用“文件头说明 + 主 section 说明”的详细注释写法，不得只留下字段名或模糊短句让读者自行猜测层级含义。
- `Code` 层负责实现；`Evidence` 层负责保存生成产物、测试结果、运行样例与验证证据。
- 允许的真相流向仅为：`Framework -> Product Spec -> Implementation Config -> Code -> Evidence`。
- 禁止下层反向定义上层：`Code/Evidence` 不得成为框架、公理、产品真相或实现配置的第一性事实来源。
## 4. 模块分层规则（强制）

- 层级编号 `X` 必须由文件名前缀 `Lx-` 决定。
- 模块是文件级单元：每个 `framework/<module>/Lx-Mn-*.md` 文件对应一个模块。
- 当同层存在多个模块文件时，必须使用 `Lx-Mn-*.md` 文件名声明模块编号（`M0/M1/...`）。
- 模块别名（若需要）是文件级属性，不得绑定到单个 `B*`。
- 模块设计应从更基础的结构层起步，再向更高层结构展开。
- 模块间显式引用关系是结构判定的第一性事实；`Lx` 只表达当前框架内部的局部分层与阅读分组，不得替代真实引用图。
- 模块内基础节点必须可规范化表示为：`L{X}.M{n}.B{m}`。
- `B*` 必须是抽象、可复用结构，不得直接固化颜色、文案、主题值等实例参数。
- 实例参数应通过边界项表达，边界用于约束结构展开空间。
- 当基依赖本框架更低层模块时，应在基定义中直接写模块引用（例如 `L0.M0[R2,R3] + L0.M1[R2,R3]`）。
- `L0` 的 `B*` 必须保持自足，不得再引用其他模块。
- 若知识库、业务场景等领域框架需要建立在更基础的通用框架之上，可在当前框架的根层或更高层显式写外部模块引用（例如 `frontend.L0.M0[R1,R3]` 或 `frontend.L1.M0[R1,R3]`）；外部依赖是否合法，优先依据显式引用图的单向性、可解释性与无环性判断，而不是仅凭被引模块文件名中的 `L0/L1` 标签判断。
- 禁止使用 `上游模块：...` 追加字段；上游模块引用必须直接写在 `B*` 定义主句中。
- 本框架内部的显式引用必须只指向更低的局部层：若当前模块位于 `Lx`，则其本地上游只能来自 `Ly` 且 `y < x`；是否必须相邻由框架总体规划决定，不再单独作为硬约束。
- 禁止本框架内部出现同层或反向引用：`Lx -> Lx`、`Lx -> Ly (y > x)` 均无效。
- 显式引用图必须无环；若存在循环依赖，应先重构结构，再讨论层级标签。
- 更高层领域通用框架可以建立在更基础的通用框架之上，但依赖方向必须单向，且不得把更基础通用结构和本领域结构混成同一节点责任。

## 5. 树化映射规则（强制）

- `mapping/mapping_registry.json` 的 `tree` 字段仅用于表达 L0-L3 框架标准树。
- `mapping/mapping_registry.json` 的 `tree` 是标准树的机器同步视图，应与 `uv run python scripts/sync_mapping_registry.py` 的生成结果保持一致，不再作为独立人工真相源。
- `projects/` 是实例层，不得挂入 `mapping/mapping_registry.json` 的 `tree`。
- `projects/` 只允许保存实例配置、说明文档、品牌资源与生成产物；项目实例不得在 `projects/<project_id>/` 下直接存放手写实现代码。
- `product_spec.toml` 的顶层 section 必须与被实例化框架的大类边界保持稳定映射；禁止引入游离杂项 section。
- 被实例化的边界必须能追溯到 `product_spec.toml` 中的主 section，并可定位到对应配置落点。
- 所有“标准 -> 实现”的一一对应关系，必须挂载在 `mapping/mapping_registry.json` 的 `mappings` 字段。
- `tree` 必须具备稳定 `id`，并使用 `children` 维护父子关系。
- `tree` 中每个 `kind=file` 节点必须指向唯一文件，禁止重复挂载同一文件。
- `tree` 禁止挂载工程执行文件或实现证据文件（`AGENTS.md`、`src/*`、`docs/*` 等）。
- `L0/L1` 文件必须位于 `specs/`。
- `L2` 文件必须位于 `framework/<module>/L2-Mn-*.md`。
- `L3` 注册文件必须位于 `mapping/`。

## 6. 变更传导规则（强制）

- 当 L0 变更时，L1/L2/L3 必须同步评估并按需更新。
- 当 L1 变更时，L2/L3 必须同步评估并按需更新。
- 当 L2 变更时，L3 必须同步更新。
- 当 L3 变更时，必须执行反向验证，确认不违反 L0/L1/L2。
- 模块层级文档变更时，必须通过模块分层校验。

## 7. 验证机制

机器可读映射：`mapping/mapping_registry.json`

验证命令：
```bash
uv run python scripts/materialize_project.py
uv run python scripts/validate_strict_mapping.py
uv run python scripts/validate_strict_mapping.py --check-changes
```

## 8. 评审准入门槛

合并前至少满足：
- 标准树关系未被破坏
- 映射验证通过
- 模块分层与组合关系验证通过
- 变更方向满足传导规则
