# 规范总纲与树形结构（L0）

## 1. 目的
本文件是仓库规范总纲，定义标准树、模块分层目录约束和变更传导规则。

## 2. 标准树（强制）

标准文件目录采用“职责分离”：
- `specs/`：承载 L0/L1 抽象规范
- `framework/<module>/Lx-Mn-*.md`：承载领域模块分层标准
- `mapping/`：承载机器可读映射注册（L3）

## 3. 模块分层规则（强制）

- 层级编号 `X` 必须由文件名前缀 `Lx-` 决定。
- 模块是文件级单元：每个 `framework/<module>/Lx-Mn-*.md` 文件对应一个模块。
- 当同层存在多个模块文件时，必须使用 `Lx-Mn-*.md` 文件名声明模块编号（`M0/M1/...`）。
- 模块别名（若需要）是文件级属性，不得绑定到单个 `B*`。
- 模块设计应从 `L0` 抽象结构层起步，再向高层展开。
- 模块内基础节点必须可规范化表示为：`L{X}.M{n}.B{m}`。
- `B*` 必须是抽象、可复用结构，不得直接固化颜色、文案、主题值等实例参数。
- 实例参数应通过边界项表达，边界用于约束结构展开空间。
- 当基依赖上游模块时，应在基定义中直接写模块引用（例如 `L0.M0[R2,R3] + L0.M1[R2,R3]`）。
- `L0` 的 `B*` 必须保持自足，不得再引用其他模块。
- 禁止使用 `上游模块：...` 追加字段；上游模块引用必须直接写在 `B*` 定义主句中。
- 组合边仅允许：`Lx.M* -> L(x+1).M*`（下游到上游）。
- 禁止同层组合：`Lx -> Lx`。
- 禁止跨层跳级组合：`Lx -> L(x+2)`、`L(x+1) -> Lx` 等反向/跳级关系。

## 4. 树化映射规则（强制）

- `mapping/mapping_registry.json` 的 `tree` 字段仅用于表达 L0-L3 框架标准树。
- 所有“标准 -> 实现”的一一对应关系，必须挂载在 `mapping/mapping_registry.json` 的 `mappings` 字段。
- `tree` 必须具备稳定 `id`，并使用 `children` 维护父子关系。
- `tree` 中每个 `kind=file` 节点必须指向唯一文件，禁止重复挂载同一文件。
- `tree` 禁止挂载工程执行文件或实现证据文件（`AGENTS.md`、`src/*`、`docs/*` 等）。
- `L0/L1` 文件必须位于 `specs/`。
- `L2` 文件必须位于 `framework/<module>/L2-Mn-*.md`。
- `L3` 注册文件必须位于 `mapping/`。

## 5. 变更传导规则（强制）

- 当 L0 变更时，L1/L2/L3 必须同步评估并按需更新。
- 当 L1 变更时，L2/L3 必须同步评估并按需更新。
- 当 L2 变更时，L3 必须同步更新。
- 当 L3 变更时，必须执行反向验证，确认不违反 L0/L1/L2。
- 模块层级文档变更时，必须通过模块分层校验。

## 6. 验证机制

机器可读映射：`mapping/mapping_registry.json`

验证命令：
```bash
uv run python scripts/validate_strict_mapping.py
uv run python scripts/validate_strict_mapping.py --check-changes
```

## 7. 评审准入门槛

合并前至少满足：
- 标准树关系未被破坏
- 映射验证通过
- 模块分层与组合关系验证通过
- 变更方向满足传导规则
