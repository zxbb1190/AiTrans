# 前端样式实现质量标准

## 0. 定位
本文件定义 CSS、SCSS、Less、CSS-in-JS 等前端样式代码的通用最低质量要求。

## 1. 核心要求

- 样式代码必须通过 lint、构建或等价校验，不得提交语法错误、失效选择器或无法解析的样式结构。
- 公共样式块、设计 token、变体名和状态名必须显式命名，不得依赖隐式层级或偶然选择器优先级。

- 样式代码应按“结构块 + 变体流程”组织，而不是让核心视觉结构长期散落在临时选择器和重复字面量里。
- 结构块必须自举、自成体系，像芯片一样具备稳定边界、稳定 token 和稳定语义；凡在多个组件、页面或场景之间复用的稳定视觉单元，默认应收敛为具名 token、具名样式块、具名 variant 或等价样式结构。
- 样式结构负责表达“视觉结构是什么”；状态切换和应用时机负责表达“结构何时生效、如何组合”。
- 纯局部微调可以保持内联或局部规则；但只要一组颜色、间距、字号、断点、状态样式或组件外观已经具备稳定语义边界，就必须升级为具名结构块。
- 禁止让核心视觉结构长期依赖选择器碰运气、层级偶然性或重复覆写。

- 禁止把颜色值、间距值、断点、阴影、层级和主题规则直接散落硬编码在多个位置。
- 这类信息必须进入设计 token、主题配置或统一真相源。
- 同一视觉规则不得在多个样式块重复维护。

- 禁止在样式代码、资源引用或错误信息中写入机器本地绝对路径。
- 路径必须通过相对路径、配置或运行时发现获得。

- 每个 token、variant、状态类或主题开关都必须对应明确的下游视觉效果。
- 无实际作用的 token、variant 或状态样式必须删除。
- 禁止保留只声明、不生效的死样式结构。

- 关键样式逻辑必须可测试、可校验、可回归。
- 修复缺陷或新增规则时，必须补充对应验证，防止问题再次出现。

## 2. 结论

- 样式可解析
- 视觉有结构
- token 不散落
- 路径可移植
- 配置真生效
- 结果可验证
