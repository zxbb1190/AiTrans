# Python 实现质量标准

## 0. 定位
本文件定义 Python 代码的通用最低质量要求。

## 1. 核心要求

- Python 代码必须通过静态类型检查。
- 公共函数、方法、返回值和核心数据结构必须显式标注类型。
- 不得无约束扩散 `Any`。

- 代码应按“结构块 + 变换流程”组织，而不是让核心概念长期散落在参数、裸 `dict` 或隐式约定里。
- 结构块必须自举、自成体系，像芯片一样具备稳定边界、稳定字段和稳定语义；凡在多个函数之间传递、承载核心真相或表示稳定阶段结果的结构块，默认应使用 `@dataclass` 建模。
- `@dataclass` 负责表达“这是什么结构”；`def` 负责表达“结构如何被构造、校验、变换和输出”。
- 纯计算、无状态、无稳定字段的辅助逻辑，允许保持 `def`；不得为了形式统一而使用“零字段 `@dataclass` + 一组 `@staticmethod`”伪装结构化。
- 禁止让核心结构长期以裸 `dict`、匿名 tuple 或并行变量组承载；若一组字段具备稳定语义边界，就必须收敛为具名结构块。

- 禁止把环境差异、业务规则、白名单、模板约束或可变策略直接写死在代码里。
- 这类信息必须进入配置、契约文件或统一真相源。
- 同一规则不得在多个模块重复维护。

- 禁止在代码、模板、文档生成或错误信息中写入机器本地绝对路径。
- 路径必须通过相对路径、配置或运行时发现获得。

- 每个配置项都必须对应明确的下游行为。
- 无实际作用的配置项必须删除。
- 禁止保留只声明、不生效的死配置。

- 关键逻辑必须可测试、可校验、可回归。
- 修复缺陷或新增规则时，必须补充对应验证，防止问题再次出现。

## 2. 结论

- 类型可检查
- 规则不散落
- 路径可移植
- 配置真生效
- 结果可验证
