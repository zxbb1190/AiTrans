# JavaScript 实现质量标准

## 0. 定位
本文件定义 JavaScript 代码的通用最低质量要求。

## 1. 核心要求

- JavaScript 代码必须具备可检查的静态约束，至少通过 lint；若工具链允许，应启用 `checkJs`、JSDoc 类型检查或等价机制。
- 公共函数、模块导出和核心数据结构必须具备显式 JSDoc 或等价的结构说明。
- 不得让核心输入输出长期处于“只靠运行时猜测”的状态。

- 代码应按“结构块 + 变换流程”组织，而不是让核心概念长期散落在参数、匿名对象或隐式约定里。
- 结构块必须自举、自成体系，像芯片一样具备稳定边界、稳定字段和稳定语义；凡在多个函数、组件或模块之间传递、承载核心真相或表示稳定阶段结果的结构块，默认应使用具名 `@typedef`、具名工厂、带真实状态的 `class` 或等价结构建模。
- 结构定义负责表达“这是什么结构”；函数负责表达“结构如何被构造、校验、变换和输出”。
- 纯计算、无状态、无稳定字段的辅助逻辑，允许保持函数；不得为了形式统一而使用“零字段 class + 一组 static 方法”伪装结构化。
- 禁止让核心结构长期以匿名对象字面量、裸 JSON 或并行变量组承载；若一组字段具备稳定语义边界，就必须收敛为具名结构块。

- 禁止把环境差异、业务规则、白名单、模板约束或可变策略直接写死在代码里。
- 这类信息必须进入配置、契约文件或统一真相源。
- 同一规则不得在多个模块重复维护。

- 禁止在代码、模板、构建产物说明或错误信息中写入机器本地绝对路径。
- 路径必须通过相对路径、配置或运行时发现获得。

- 每个配置项都必须对应明确的下游行为。
- 无实际作用的配置项必须删除。
- 禁止保留只声明、不生效的死配置。

- 关键逻辑必须可测试、可校验、可回归。
- 修复缺陷或新增规则时，必须补充对应验证，防止问题再次出现。

## 2. 结论

- 结构可检查
- 结构有名字
- 规则不散落
- 路径可移植
- 配置真生效
- 结果可验证
