# HTML 与模板实现质量标准

## 0. 定位
本文件定义 HTML、JSX、模板片段等前端模板代码的通用最低质量要求。

## 1. 核心要求

- 模板代码必须通过解析、构建或等价校验，不得提交结构不闭合、语义不合法或运行时才能发现的基础错误。
- 公共模板入口、组件插槽、区域边界和关键属性必须显式命名，不得依赖隐式层级猜测。

- 模板应按“结构块 + 组合流程”组织，而不是让核心界面结构长期散落在重复片段和临时拼装里。
- 结构块必须自举、自成体系，像芯片一样具备稳定边界、稳定槽位和稳定语义；凡在多个页面、组件或场景之间复用的稳定界面单元，默认应收敛为具名组件、具名 partial、具名 slot 结构或等价模板块。
- 模板块负责表达“界面结构是什么”；脚本或数据层负责表达“结构如何取值、校验和变换”。
- 纯展示片段可以保持局部内联；但只要某段模板已经承载稳定区域语义、稳定属性集合或重复组合关系，就必须升级为具名结构块。
- 禁止在模板中长期混入复杂业务判断、规则推导和数据整形；模板应优先承载结构，不应成为业务真相的第一性来源。

- 禁止把环境差异、业务规则、白名单、模板约束或可变策略直接写死在模板里。
- 这类信息必须进入配置、契约文件、数据模型或统一真相源。
- 同一规则不得在多个模板片段重复维护。

- 禁止在模板、资源引用或错误信息中写入机器本地绝对路径。
- 路径必须通过相对路径、配置或运行时发现获得。

- 每个模板配置项、slot、属性或 data hook 都必须对应明确的下游行为。
- 无实际作用的属性、slot 或模板开关必须删除。
- 禁止保留只声明、不生效的死结构。

- 关键模板逻辑必须可测试、可校验、可回归。
- 修复缺陷或新增规则时，必须补充对应验证，防止问题再次出现。

## 2. 结论

- 结构可解析
- 区域有名字
- 逻辑不漂移
- 路径可移植
- 配置真生效
- 结果可验证
